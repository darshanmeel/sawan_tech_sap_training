Overview

Scenario

Your finance team needs ACDOCA data for year-end close analytics. You will extract one company code, one fiscal year — enough for meaningful analysis without hitting system limits. This walkthrough focuses on the SAP-side checks: confirming delta capability, monitoring ODP subscriptions, and understanding the first-delta behavior that surprises most teams on their first run.

Estimated time: 90 minutes.

What you have confirmed

You have verified that I_JournalEntryItem supports delta extraction on your system, understand that the first ODP delta returns 0 records (by design), and have a reconciliation baseline. When you need more years or company codes, the Expert walkthrough scales these patterns via SLT.

Before you start

License check

ODP extraction via OData is permitted under Runtime and Full Use licenses. License trap: Runtime vs Full Use →

ODP (Operational Data Provisioning) is permitted under both Runtime and Full Use SAP licenses when you access data through published CDS views via the OData API.

SAP Note 3255746 (Feb 2024): Third-party tools that call ODP via RFC (ADF SAP CDC, Qlik, Fivetran ODP, Informatica ODP, etc.) are no longer permitted. Use ODP via OData, or switch to SLT. Verify with your licensing contact. Full note (SAP login required).

SLT (SAP Landscape Transformation) requires a Full Use SAP license. It is not permitted under Runtime licenses. Verify in SLICENSE that "SLT Landscape Transformation" is listed as Active, Type = Full Use before you configure anything in LTCO.

License trap: Runtime vs Full Use →

Direct RFC extraction (e.g., RFC_READ_TABLE) is generally permitted for read-only access under Runtime and Full Use licenses, but is not recommended for production use — it bypasses authorization checks that released CDS views enforce and is subject to SAP Note 3255746 restrictions when wrapped by third-party ODP connectors.

SAP prerequisites

These steps are one-time per SAP system (not repeated per table). Once your RFC user, authorizations, and RFC destination are set up, every table extraction reuses the same connection. The checks below, in order, are the difference between a smooth extraction and a multi-day incident.

  1. 01 · Licence

    Confirm your extraction method is covered by your installed SAP licence before anyone designs anything.

    How to check

    Runtime licences cover ODP through OData. SLT replication requires a Full Use licence. Ask your SAP licensing contact; do not assume. See the dedicated article on Runtime vs Full Use, and note that SAP Note 3255746 prohibits third-party tools from using ODP via RFC — compliant paths are ODP over OData or SLT.

  2. 02 · Released CDS view

    Every table covered here has an extraction-safe released CDS view; confirm it exists before building.

    How to check

    Open the view in SE80 (or in ADT) and confirm the header annotation @Analytics.dataExtract: true is present. If it is missing, the view is not extractable — raw-table reads are not the fix.

  3. 03 · Authorisations

    A System-type technical user with the minimum authorisation objects for your chosen method — no named users, no wildcards.

    How to check

    Create a System user via SU01 and assign a role via PFCG. For ODP: S_RFC (RFC_TYPE=FUNC) and S_ODP_READ (ODPSOURCE=ABAP_CDS). For SLT your Basis team also needs authorisation in LTCO and LTRS. For RFC direct add S_TABU_DIS scoped to the table's authorisation group — never wildcard it.

  4. 04 · Row-count baseline

    Use SE16N to measure the intended partition before extraction — you cannot reconcile without this number.

    How to check

    Open SE16N, apply the same filter your extraction will use, and record the count. Under 100M rows is usually safe for a single ODP partition; larger volumes need further splitting or SLT. A count in the billions means the filter is misconfigured — rethink the partition before running anything.

Also confirm

  • Technical extraction user (System type) created in SU01 with S_RFC and S_ODP_READ authorizations
  • SAP license type confirmed (Runtime or Full Use) — affects which extraction method is permitted

SAP side

Steps to complete in your SAP system. Use the Method picker above to switch between ODP, SLT, and RFC approaches.

0 of 5 steps complete
  1. Step 1

    Confirm ACDOCA partition size in SE16N

    Before running any extraction, use SE16N to count rows for your intended partition. Filter on RBUKRS (company code) and RYEAR (fiscal year). If the count exceeds 500M rows for a single partition, move to the Expert walkthrough — that volume requires SLT or sub-partition strategies not covered here.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → ACDOCA → Filter RBUKRS='1000' AND RYEAR=2024 → Count.
    Verify
    Row count is below 500M. Write it down for reconciliation.
  2. Step 2

    Confirm I_JournalEntryItem CDS view and delta annotation

    In SE80, open I_JournalEntryItem and inspect the annotations. For delta extraction you need @Analytics.dataExtraction.delta.changeDataCapture.automatic: true. If that annotation is absent, the view supports only full-load extraction — delta will not work.

    Do this in SAP
    Open transaction SE80 (opens in new tab) and navigate to Repository Browser → I_JournalEntryItem → Annotations.
    Verify
    I_JournalEntryItem has @Analytics.dataExtract: true. Check for the delta changeDataCapture annotation. If absent, confirm with your Basis team whether a support package is needed to enable it.
    Why this matters

    If you run a delta extraction against a view that lacks the CDC annotation, ODP returns all rows on every run — the same behavior as a full load but without you realising it. This leads to duplicate data in your target.

  3. Step 3

    Monitor the ODP subscription in ODQMON

    After your extraction tool registers, open ODQMON and navigate to the ABAP_CDS context. Find the subscription for I_JournalEntryItem. Note: after an init-delta run, the first delta request returns 0 records. This is correct ODP behavior — the delta mechanism captures changes from the init point forward, not from before it. SAP Note 2884410 documents this.

    Do this in SAP
    Open transaction ODQMON (opens in new tab) and navigate to Subscriptions → Context: ABAP_CDS → I_JournalEntryItem.
    Verify
    Subscription for I_JournalEntryItem appears. After init, delta queue shows 0 pending records — this is expected.
  4. Step 4

    Reconcile extracted row count against SE16N

    After the full-load extraction completes, return to SE16N and recount ACDOCA rows for the same RBUKRS and RYEAR partition. The count in your target system should match within a small window for in-flight postings.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → ACDOCA → Filter RBUKRS='1000' AND RYEAR=2024 → Count.
    Verify
    Target row count matches SE16N count within a tolerable margin for in-flight transactions.
  5. Step 5

    Add posting period filter if the partition is still too large

    If SE16N shows your RBUKRS+RYEAR partition is too large (e.g., 200M+ rows) for a safe ODP extraction, add a posting period filter. In SE16N, filter on POPER (posting period, values 001–012 for months) to count rows per quarter. Then extract one quarter at a time.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → ACDOCA → Filter RBUKRS='1000' AND RYEAR=2024 AND POPER='001' → Count.
    Verify
    Each quarterly partition (e.g., POPER='001' through '003') has a manageable row count. Adjust partition granularity accordingly.
  1. Step 1

    Confirm Full Use license and SLT system access

    SLT requires a Full Use license — it reads ACDOCA directly, bypassing the CDS authorization layer. Confirm license with your SAP Basis team before proceeding. Then confirm you have access to transaction LTRC on the SLT system (which may be a separate SAP system from the source ERP).

    Do this in SAP
    Open transaction LTRC (opens in new tab) and navigate to SLT system → LTRC → Check mass transfer list.
    Verify
    LTRC opens without authorization error. Confirm with Basis that Full Use license is active.
  2. Step 2

    Create a Mass Transfer ID in LTRC

    In LTRC, create a new mass transfer configuration. Select your source system (the ERP where ACDOCA lives) and your target system (Kafka topic, database, or secondary HANA). Give the mass transfer a descriptive name (e.g., MT_ACDOCA_PROD). SLT will use this ID to track all replication state for ACDOCA.

    Do this in SAP
    Open transaction LTRC (opens in new tab) and navigate to LTRC → New → Mass Transfer → Source: ERP system → Target: your destination.
    Verify
    Mass Transfer ID created and visible in LTRC overview.
  3. Step 3

    Add ACDOCA as a replication object with partitioning

    Inside the mass transfer, add ACDOCA as a replication table. In the partitioning settings, define partition keys: RBUKRS (company code) and GJAHR (fiscal year). This is the most important configuration step — without partitioning, SLT reads the entire table in a single pass, which will overwhelm both SAP and the target. Each RBUKRS+GJAHR combination becomes an independent partition that LTRS can replicate in parallel.

    Do this in SAP
    Open transaction LTRC (opens in new tab) and navigate to Mass Transfer → Table Settings → Add Table: ACDOCA → Partitioning: RBUKRS + GJAHR.
    Verify
    ACDOCA appears in table list with partition keys RBUKRS and GJAHR defined.
  4. Step 4

    Configure parallel readers in LTRS

    Navigate to transaction LTRS (Replication Server) and set the number of parallel reader processes for your mass transfer. Start with 4 parallel readers. Each reader handles one partition (one RBUKRS+GJAHR combination) simultaneously. Monitor SM50 on the source ERP during the first run — if dialog processes exceed 70% utilization, reduce readers to 2 and re-test.

    Do this in SAP
    Open transaction LTRS (opens in new tab) and navigate to LTRS → Mass Transfer → Advanced Settings → Number of Parallel Jobs: 4.
    Verify
    LTRS shows 4 parallel job slots configured. SM50 on source ERP has capacity to support them.
  5. Step 5

    Start initial load and monitor in LTCO

    Activate the replication in LTRC. SLT will immediately begin the full initial load, reading each RBUKRS+GJAHR partition across your configured readers. Open transaction LTCO to monitor progress: you will see rows-per-second throughput, per-partition status (running / complete / error), and any failed partitions. Expect the initial load of a large ACDOCA (10B+ rows) to take 24–72 hours depending on partition count and reader configuration.

    Do this in SAP
    Open transaction LTCO (opens in new tab) and navigate to LTCO → Mass Transfer → ACDOCA → Initial Load Status.
    Verify
    LTCO shows partitions processing. Throughput is steady (rows/second visible). No partitions in ERROR state.
  6. Step 6

    Verify delta replication after initial load completes

    Once the initial load finishes, SLT automatically switches to delta mode — it triggers on every ACDOCA insert or update and replicates the change to the target within seconds. To verify: post a test document in the source ERP (or identify a recent posting in SE16N), then confirm it appears in the target within 60 seconds. Check LTCO for delta lag — it should show near-zero pending records under normal conditions.

    Do this in SAP
    Open transaction LTCO (opens in new tab) and navigate to LTCO → Mass Transfer → ACDOCA → Delta Status → Pending Records.
    Verify
    Delta lag in LTCO shows < 100 pending records. Test document posted in ERP appears in target within 60 seconds.
  7. Step 7

    Reconcile row count against SE16N

    After initial load completes, run a count in SE16N for each RBUKRS+GJAHR partition and compare to the count in your target system. Use the same filters: RBUKRS=1000, GJAHR=2024. Small discrepancies of a few hundred rows are normal (in-flight postings during the load window). A discrepancy of thousands of rows indicates a missed partition or a stuck reader — check LTCO for ERROR partitions and re-trigger them.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to SE16N → ACDOCA → Filter RBUKRS='1000' AND GJAHR=2024 → Count.
    Verify
    Target row count matches SE16N within a 0.01% tolerance. Re-trigger any ERROR partitions in LTCO.
Troubleshooting
ODP extraction times out after 2 hours

The partition is more than 500M rows. Use the Expert walkthrough which covers SLT parallel readers for ACDOCA at that scale.

First delta run returns 0 records — is this a bug?

No. This is documented ODP behavior per SAP Note 2884410. After an init-delta, the first delta request returns 0 records because no changes have occurred since the init point. Subsequent delta runs will return only changed records.

ODQMON shows subscription stuck in 'FETCHING' state

The extraction process is still running or has hung. Check SM50 for active ABAP processes owned by your extraction user. If the process is not active, the subscription may be corrupted. Reset it in ODQMON and restart the extraction.

Related SAP Note (opens in new tab)

+ Handling custom Z columns

ODP — Extending a CDS view with ABAP Developer

Use EXTEND VIEW ENTITY in ABAP Development Tools (Eclipse/ADT) to add Z columns without modifying the original view. The base view must carry the header annotation @AbapCatalog.viewEnhancementCategory: [#PROJECTION_LIST] — if the owner restricted data sources, only tables listed under @AbapCatalog.extensibility.dataSources can be joined.

  1. Open ADT (Eclipse). In the Project Explorer, locate the data definition of the CDS entity you want to extend.
  2. Right-click the data definition → New Data Definition. Choose your package, enter a name (e.g., Z_EXT_MY_VIEW), and add a description.
  3. Click Next, select your transport request, then choose the Extend View Entity template.
  4. Define your custom Z fields in the element list — pull from the underlying table alias or define a new association.
  5. Activate the view extension (Ctrl + F3). The base view now includes your custom elements.
extend view entity C_BaseStandardView with Z_Ext_My_View
{
  BaseTable.zz_custom_field    as ZZCustomField,
  _NewZTableAssoc
}

Because the ODP framework generates a TransientProvider at runtime from CDS metadata, Z columns added via EXTEND VIEW ENTITY automatically appear in the ODP delta queue. Refresh the schema in your tool (ADF, Databricks) to see new fields. Test locally with ABAP program RODPS_REPL_TEST.

SLT — Adding Z columns via LTRS Rule Assignment

In SLT you add and populate Z columns through Advanced Replication Settings in transaction LTRS.

  1. In your SLT system, run transaction LTRS.
  2. Under Table Settings, you can define the target table structure. To add a Z column that doesn't exist in the source, append it to the target structure here.
  3. Go to the Rule Assignment section → right-click → Add Table to add your replication table.
  4. Click the Create icon and choose either a Field-Related Rule (to populate a specific field) or an Event-Related Rule (to manipulate the whole record).
  5. Write ABAP in the Line of Code section to populate your Z column. For event-related rules use <wa_s_table>-columnname. For field-related rules use the E_ prefix: E_columnname.

RFC — Exposing Z columns via SE11 Append Structure

When extracting via RFC, Z columns added to a SAP transparent table through an Append Structure in transaction SE11 are immediately available — the RFC functions query the ABAP data dictionary dynamically.

  1. In SE11, open the transparent table, go to Extras → Append Structure, and add your Z field. Activate — the underlying database table is updated automatically.
  2. Standard RFC functions (RFC_READ_TABLE, /SAPDS/RFC_READ_TABLE2) will return the column dynamically.
  3. Explicitly request the new Z column in the FIELDS parameter of your RFC call — it is not returned unless listed there.
  4. If using a custom Function Module, update the Export table structure in the FM to include the new columns — it does not auto-discover.

Custom (Python / pyrfc)

  1. Step 1 Install dependencies

    pyrfc requires the SAP NetWeaver RFC SDK. Install it alongside the Python packages.

    pip install pyrfc boto3 pandas

# Set SDK path before importing pyrfc:
# export SAPNWRFC_HOME=/path/to/nwrfcsdk  (Linux/Mac)
# set SAPNWRFC_HOME=C:\nwrfcsdk           (Windows)

    Verify: python -c 'import pyrfc' prints no errors.

  2. Step 2 Extract ACDOCA — one company code, one fiscal year

    Always partition ACDOCA by RBUKRS (company code) and RYEAR (fiscal year). A full table read without partition crashes SAP. This example extracts company code 1000, year 2024.

    import pyrfc
import pandas as pd

conn = pyrfc.Connection(
    ashost='sap-prod.company.com',
    sysnr='00', client='100',
    user='EXTRACT_ACDOCA', passwd='<password>'
)

# Extract via I_JournalEntryItem — never raw ACDOCA
result = conn.call(
    'RODPS_REPL_ODP_EXTRACT',
    SUBSCRIBER_NAME='PYTHON_ACDOCA_2024_1000',
    SUBSCRIBER_TYPE='2',
    SUBSCRIBER_PROCESS='FULL_LOAD',
    DATA_AREA='ABAP_CDS',
    ENTITY_NAME='I_JournalEntryItem',
    MAX_RECORDS=1000000,
    FIELDS=[{'FIELDNAME': '*'}],
    SELECTION=[
        {'FIELDNAME': 'RBUKRS', 'SIGN': 'I', 'OPTION': 'EQ', 'LOW': '1000'},
        {'FIELDNAME': 'RYEAR',  'SIGN': 'I', 'OPTION': 'EQ', 'LOW': '2024'}
    ]
)

records = result.get('DATA', [])
df = pd.DataFrame(records)
print(f"Extracted {len(df)} rows — ACDOCA RBUKRS=1000, RYEAR=2024")

    Verify: Row count matches SE16N for RBUKRS=1000, RYEAR=2024. Never run without the RBUKRS+RYEAR filter.

  3. Step 3 Write to S3 as Parquet

    Use Parquet, not CSV, for ACDOCA. At 5–50M rows per partition, CSV is too slow to load downstream. Parquet with snappy compression reduces size by 70–80%.

    import boto3
import pyarrow as pa
import pyarrow.parquet as pq
from io import BytesIO

s3 = boto3.client('s3', region_name='eu-west-1')
table = pa.Table.from_pandas(df)
buf = BytesIO()
pq.write_table(table, buf, compression='snappy')
buf.seek(0)
s3.put_object(
    Bucket='my-sap-landing-bucket',
    Key='sap/acdoca/rbukrs=1000/ryear=2024/acdoca.parquet',
    Body=buf.getvalue()
)
print(f"Loaded {len(df)} rows to S3 as Parquet")

    Verify: Parquet file appears in S3. Row count matches extraction count.

  2. Step 2 Write as Delta table (recommended), Iceberg table, or Parquet files

    Choose your write format based on your downstream sink. Delta is the default for Databricks-to-Databricks flows. Iceberg is the best choice when downstream consumers include BigQuery, Athena, or non-Databricks engines — it is an open standard. Parquet files are the most portable option for any sink that reads from ADLS or S3.

    # ── Option 1: Delta table (default — best for Databricks lakehouse) ──
df.write.format('delta').mode('overwrite') \
    .partitionBy('RBUKRS', 'RYEAR') \
    .option('replaceWhere', "RBUKRS='1000' AND RYEAR='2024'") \
    .saveAsTable('sap_landing.raw.acdoca')

# ── Option 2: Iceberg table (best for cross-engine / open lake) ──
# df.write.format('iceberg').mode('overwrite') \
#     .partitionBy('RBUKRS', 'RYEAR') \
#     .saveAsTable('sap_landing.raw.acdoca')

# ── Option 3: Parquet files to ADLS / S3 (most portable) ──
# df.write.format('parquet').mode('overwrite') \
#     .partitionBy('RBUKRS', 'RYEAR') \
#     .option('compression', 'snappy') \
#     .save('abfss://sap-landing@account.dfs.core.windows.net/acdoca/')
#   # or for S3: .save('s3://my-bucket/sap/acdoca/')

    Verify: Delta: sap_landing.raw.acdoca exists with partition RBUKRS=1000/RYEAR=2024. Iceberg: same table path in Iceberg catalog. Parquet: files visible in ADLS/S3 landing path.

Troubleshooting
ADF pipeline runs but no data arrives in sink

Check that the Integration Runtime can reach both the SAP host and the sink. Confirm the source ODP dataset has a valid subscription key — open ODQMON on the SAP side and verify a subscription for your OData consumer appears. Also confirm the pipeline activity timeout is at least 4 hours for heavyweight tables.

Databricks JDBC read hangs or times out

RFC_READ_TABLE via JDBC is limited to ~500 MB result sets per call by default. For large tables, enable partition mode in the JDBC options (partitionColumn, lowerBound, upperBound, numPartitions) to split the read across parallel ABAP work processes. Check SM50 on the SAP side for stuck RFC work processes.

Fivetran sync paused with "schema drift" error

Fivetran detects column changes automatically but pauses the sync to avoid data loss. Go to the connector settings, review the new/removed columns, and resume. If the column was added to a custom Append Structure, confirm it appears in SE11 before resuming — Fivetran re-reads the schema from ABAP.

Airbyte stream returns 0 records on first sync

The ODP provider may require an init-delta run before returning data. In the Airbyte connection settings, trigger a full refresh for the first run. After the init completes, subsequent incremental syncs will return changes correctly. Confirm the ODP subscription appears in ODQMON after the first run.

Custom RFC call returns truncated rows

RFC_READ_TABLE row limit is 512 bytes per row by default. Columns with long text fields overflow. Switch to /SAPDS/RFC_READ_TABLE2 or a custom BAPI that returns the full row width. Alternatively, select fewer columns per call and join results in your pipeline.

Azure Data Lake Storage Gen2

ADLS is your landing zone — your tool writes Parquet, Delta, or Iceberg files here. Downstream engines (Synapse, Databricks, Power BI) read directly from these files.

Native SAP connector option

If your organisation has an ADF SAP connector license (SAP connector for Azure Data Factory), ADF writes directly to ADLS from SAP — no intermediate tool pipeline code needed. Confirm connector licensing with your Azure team before setting up the manual pipeline below.

Before your first extraction

  • Create an ADLS Gen2 storage account and container (e.g., sap-landing).
  • Register an Entra ID (Azure AD) app. Grant it Storage Blob Data Contributor on the container. Store tenant ID, client ID, and secret in your pipeline's secrets manager.
  • Agree on a partition path convention: sap/acdoca/rbukrs=1000/year=2024/. Partition columns must match the filter dimensions used in the SAP extraction.
  1. 1. Confirm the file landed after extraction

    After your tool runs, confirm the Parquet (or Delta/Iceberg) file lands in the container at the correct partition path. Use Azure Storage Explorer or the Azure portal. Prefer Parquet with Snappy compression; use Delta or Iceberg if your tool supports table formats.

    Verify: File visible at sap/acdoca/rbukrs=1000/year=2024/ in the container.

  2. 2. Validate row count from Synapse or Databricks

    Run a count query directly against the landed files. Compare to the SE16N count from the SAP-side steps. This is your reconciliation checkpoint before handing data to analytics.

    Verify: SELECT COUNT(*) FROM parquet.`abfss://sap-landing@account.dfs.core.windows.net/sap/acdoca/...` matches SE16N.

Amazon S3

S3 is your landing zone — your tool writes Parquet files here. Downstream engines (Athena, Redshift Spectrum, Databricks on AWS) read directly from S3, or you COPY INTO Snowflake/Redshift from here.

Native SAP connector option

If your organisation has AWS AppFlow with SAP OData connector or a licensed SAP Integration Suite adapter, AppFlow can land data directly in S3 without tool pipeline code. Check your AWS and SAP BTP licensing before building the pipeline below.

Before your first extraction

  • Create an S3 bucket in your preferred region. Enable server-side encryption (SSE-S3 minimum) and block all public access.
  • Create an IAM role with s3:PutObject and s3:GetObject on the specific bucket/prefix ARN. Use instance profiles or IRSA — avoid long-lived access keys.
  • Agree on a landing prefix convention: sap/acdoca/rbukrs=1000/year=2024/.
  1. 1. Confirm files land and register in Glue

    After extraction, confirm files appear at the S3 prefix. Register the prefix as a Glue table pointing to Parquet files so Athena and Spark can discover it. If the next layer is Snowflake, create an external stage on this bucket path.

    Verify: aws s3 ls s3://your-bucket/sap/acdoca/ shows landed files. Glue crawler ran and created the table.

  2. 2. Validate row count in Athena

    Run SELECT COUNT(*) FROM your_glue_table in Athena. Compare to SE16N. Athena scans Parquet efficiently if partition columns match the Hive-style path used for landing.

    Verify: Athena count matches SE16N. Downstream team can query the Glue table without moving files.

Google Cloud Storage (GCS)

GCS is your landing zone — your tool writes Parquet or Iceberg files here. BigQuery reads them directly as an external table or via a load job. No intermediate copy needed.

Native SAP connector option

If your organisation has the Google Cloud BigQuery Connector for SAP (available via SAP BTP / SAP LT Replication Server), it streams data directly from SAP ABAP tables into BigQuery without any intermediate GCS files or pipeline code. Confirm with your SAP Basis and GCP teams whether this connector is licensed before building the file-based pipeline below.

Before your first extraction

  • Create a GCS bucket in your preferred region. Enable uniform bucket-level access and Object Versioning.
  • Create a GCP service account. Grant it Storage Object Admin on the bucket. Store the JSON key in Secret Manager.
  • Set a landing prefix: sap/acdoca/rbukrs=1000/year=2024/.
  1. 1. Create BigQuery external table or load job

    After landing files in GCS, create a BigQuery external table (zero-copy, always current) or run a load job (faster queries, persistent storage). For Iceberg landing, BigQuery supports Iceberg tables via GCS BigLake — gives full DML and snapshot history.

    Verify: Table sap_landing.acdoca exists in BigQuery. Schema matches SE11.

  2. 2. Validate row count

    Run SELECT COUNT(*) FROM sap_landing.acdoca. Compare to SE16N. Partition by posting date or extraction date to keep BigQuery query costs manageable at scale.

    Verify: Count matches SE16N. Partition metadata visible in BigQuery console.

Snowflake

Snowflake reads Parquet files your tool landed in ADLS or S3. Two loading patterns: COPY INTO (batch, scheduled) or Snowpipe (continuous, file-triggered — runs automatically when a file lands, no scheduling needed).

Native SAP connector option — skip the steps below if licensed

If your organisation has a Snowflake Native SAP BDC Connector license, Snowflake connects directly to SAP — no tool pipeline code, no external stage, and no Parquet files needed. The connector extracts and loads natively. If this applies to you, configure the connector in Snowflake Partner Connect with your SAP RFC credentials and skip the COPY INTO / Snowpipe steps below.

Before your first extraction (file-based approach)

  • Create a dedicated database (SAP_LANDING), schema (RAW), and warehouse (SAP_LOAD_WH — X-Small for bulk loads).
  • Create a Snowflake storage integration (not access keys) for your ADLS container or S3 bucket. Create an external stage pointing to the landing path where your tool drops Parquet files.
  • Create the target table: CREATE TABLE SAP_LANDING.RAW.acdoca ... with columns matching SE11. Or use COPY INTO with MATCH_BY_COLUMN_NAME to auto-map Parquet columns.
  1. 1. Load from external stage — COPY INTO or Snowpipe

    Batch (COPY INTO): Run COPY INTO SAP_LANDING.RAW.acdoca FROM @your_stage/sap/acdoca/ PATTERN='.*parquet' FILE_FORMAT=(TYPE=PARQUET). Schedule via a Snowflake task or your orchestrator after each extraction.

    Continuous (Snowpipe): Create a pipe: CREATE PIPE sap_acdoca_pipe AS COPY INTO SAP_LANDING.RAW.acdoca FROM @your_stage/sap/acdoca/ FILE_FORMAT=(TYPE=PARQUET). Enable auto-ingest on the stage — Snowpipe fires automatically when a file lands. No scheduling required; ideal for near-real-time SLT feeds.

    Verify: LIST @your_stage/sap/acdoca/ shows landed files. COPY INTO or Snowpipe status shows zero skipped files.

  2. 2. Validate row count after load

    Run SELECT COUNT(*) FROM SAP_LANDING.RAW.acdoca. Compare to SE16N row count from the SAP-side steps. Check INFORMATION_SCHEMA.LOAD_HISTORY for skipped or errored files if counts diverge.

    Verify: SELECT COUNT(*) matches SE16N. Zero error rows in load history.

BigQuery

BigQuery reads Parquet or Iceberg files from GCS, or accepts direct writes from Fivetran and ADF. If your tool wrote to GCS as Parquet, create an external table — no copy needed.

Native SAP connector option — skip the steps below if licensed

Confirm with your SAP Basis team. If licensed, skip the file-based steps below.

  • Create a BigQuery dataset (e.g., sap_landing) in the same region as your GCS bucket. Cross-region reads from GCS are not supported.
  • Grant your service account BigQuery Data Editor on the dataset and Storage Object Viewer on the GCS bucket (for external tables).
  1. 1. Create external table, Iceberg table, or use direct connector

    Parquet path: CREATE EXTERNAL TABLE sap_landing.acdoca OPTIONS (format='PARQUET', uris=['gs://your-bucket/sap/acdoca/*']) — zero-copy, always current.
    Iceberg path: Register as a BigLake Iceberg table for full DML support and snapshot history.
    Direct connector (Fivetran/ADF): Writes a native BigQuery table directly — no staging or external table needed.

    Verify: Table sap_landing.acdoca exists and schema matches SE11.

  2. 2. Validate row count

    Run SELECT COUNT(*) FROM sap_landing.acdoca. Compare to SE16N. Partition the table by posting date or extraction date to keep per-query costs manageable at scale.

    Verify: Count matches SE16N. Partition metadata visible in BigQuery console.

Microsoft Fabric / OneLake

Fabric has three file-based paths: (1) ADF writes to Lakehouse via Copy Activity, (2) Databricks writes Delta/Iceberg to ADLS and Fabric shortcuts it, (3) your tool lands Parquet in ADLS and Fabric shortcuts the folder as an external table.

Native SAP connector option — skip the steps below if licensed

If your organisation has the Microsoft Fabric SAP CDC connector or an SAP integration via Azure Logic Apps with native Fabric write support, data flows directly from SAP into OneLake without any intermediate files or tool pipeline code. Confirm with your Microsoft and SAP teams. If licensed, skip the file-based steps below.

Before your first extraction (file-based approach)

  • Create a Lakehouse in your Fabric workspace (e.g., SAP_Landing). Create a Tables/acdoca/ folder for Delta/Iceberg tables, or Files/acdoca/ for raw Parquet.
  • Decide your ingestion path before extracting: ADF Copy Activity, Databricks ADLS shortcut, or Parquet ADLS shortcut. Each wires up differently — agree with the analytics team first.
  1. 1. Set up your ingestion path

    ADF pipeline: Copy Activity to Lakehouse sink writes directly — no intermediate files needed.
    Databricks Delta/Iceberg: Create an OneLake shortcut pointing to the ADLS path where Databricks wrote. Fabric auto-registers as a Delta table in the SQL endpoint.
    Parquet from any tool: Create an OneLake shortcut to the ADLS Parquet folder. Register as an external table in the Lakehouse.

    Verify: Data appears in the Lakehouse explorer after a test extraction run.

  2. 2. Validate in SQL endpoint

    Open the Lakehouse SQL endpoint in Fabric. Run SELECT COUNT(*) FROM acdoca. Fabric automatically registers Delta tables in the SQL endpoint — Parquet external tables appear under Files.

    Verify: Row count matches SE16N. Table visible in Lakehouse explorer.

Databricks Lakehouse

If your tool was Databricks, the Delta or Iceberg table is already written — querying it IS the sink step. If another tool wrote Parquet to ADLS/S3, register it as an external Delta or Iceberg table in Unity Catalog.

Native SAP connector option — skip the steps below if licensed

If your organisation has the Databricks SAP Connector (SAP LT Replication + Databricks native integration), it writes SAP change data directly to a Delta table in real time — no tool pipeline code or external files needed. Before your first extraction (file-based approach)

  • Ensure Unity Catalog is enabled. Create a catalog (e.g., sap_landing) and schema (raw).
  • If Databricks is your extraction tool: the saveAsTable call in the Tool tab writes the table — no further sink setup needed.
  • If another tool writes Parquet to ADLS/S3: note the exact file path for registering as an external table.
  1. 1. Confirm Delta/Iceberg table or register Parquet

    If Databricks was your tool: Run SHOW TABLES IN sap_landing.raw — the table should already exist.
    If tool wrote Parquet: Register as external: CREATE TABLE sap_landing.raw.acdoca USING PARQUET LOCATION 'abfss://...'
    Or convert zero-copy to Delta: CONVERT TO DELTA parquet.`abfss://...` — gives time-travel and ACID transactions on the existing files.

    Verify: SELECT COUNT(*) FROM sap_landing.raw.acdoca returns the expected row count.

  2. 2. Validate count and grant downstream access

    Compare the row count to SE16N. Then grant SELECT to the analytics team: GRANT SELECT ON TABLE sap_landing.raw.acdoca TO `analytics-group`. Downstream users should not need extraction credentials.

Other landing zone

Whatever your sink: it receives Parquet, Delta, or Iceberg files from your tool. Ensure your downstream system can read one of these formats efficiently before landing data in it.

Before your first extraction

  • Confirm your sink can read Parquet (most portable), Delta, or Iceberg. Avoid CSV for tables over 10M rows.
  • Agree on a partitioned path convention with the analytics team: sap/acdoca/rbukrs=1000/year=2024/run=TIMESTAMP/.
  • Test a small write to your sink before running a full extraction — confirm schema mapping, encoding, and permissions.
  1. 1. Confirm file lands at the correct path

    After extraction, verify the file appears at the agreed partition path. Check file size is reasonable for the expected row count (Parquet with Snappy is typically 100–300 bytes per row for journal data).

    Verify: File is present and non-zero. Schema is readable by your downstream tool.

  2. 2. Row-count reconciliation is mandatory

    Always compare the row count from your sink against the SE16N count from the SAP-side steps. A mismatch means the extraction dropped rows or the filter scopes differ — investigate before handing data to analytics. Automate this check in your pipeline so it fails the run on divergence.

    Verify: Your pipeline has an automated row-count check that fails the run if counts diverge beyond a tolerance threshold.

Troubleshooting
Files land in ADLS but Synapse/Databricks can't read them

Confirm the storage account has Hierarchical Namespace (HNS) enabled — standard ADLS Blob storage doesn't support ABFSS URIs. Check that the service principal has Storage Blob Data Reader on the container, not just Contributor on the storage account. Verify the Parquet file isn't corrupted by running a quick count from Azure Storage Explorer's preview.

S3 upload succeeds but Glue crawler shows 0 records

The Parquet file may be in a nested folder that doesn't match the Glue partition pattern. Confirm the S3 prefix uses Hive-style partitions (e.g., rbukrs=1000/year=2024/). Run MSCK REPAIR TABLE your_table in Athena after re-crawling to register new partitions. Check that the Glue IAM role has s3:GetObject on the exact bucket ARN, not just the bucket root.

COPY INTO shows 0 rows loaded or "file format mismatch"

Confirm the external stage points to the correct S3/ADLS prefix — run LIST @your_stage/sap/slug/ and verify files appear. If using MATCH_BY_COLUMN_NAME, the Parquet column names must exactly match the table column names (case-sensitive). For Snowpipe, check pipe status with SELECT SYSTEM$PIPE_STATUS('your_pipe') and inspect the error log table.

BigQuery external table returns stale or missing data

External tables in BigQuery always reflect the current state of the underlying GCS files — if new files were added after the external table was created, they are automatically included (no refresh needed). If data appears stale, confirm the GCS bucket path in the external table definition matches where your tool is writing. For persistent tables loaded via a load job, re-run the load after each extraction.

Fabric Lakehouse shortcut shows empty or stale data

OneLake shortcuts reflect ADLS in near-real time but may cache metadata. Click "Refresh" on the Lakehouse table in the Fabric UI. If the Delta table shows stale data, run DESCRIBE HISTORY your_table in a Fabric notebook to confirm the latest Delta commit was written. For Parquet shortcuts, the file must be at exactly the path the shortcut points to — a new subfolder will not be automatically included.

Databricks row count doesn't match SE16N after writing

Check for concurrent writes — if another pipeline wrote to the same Delta table path during the extraction, rows may be duplicated or the table may be in an inconsistent state. Run DESCRIBE HISTORY sap_landing.raw.slug to see all commits. Use VACUUM followed by RESTORE if you need to roll back. For initial loads, use OVERWRITE mode to ensure idempotency.

BigQuery external table on GCS returns permission denied

The BigQuery service account needs Storage Object Viewer on the GCS bucket — not on the project. Add the IAM binding at the bucket level, not the project level. If using a BigLake Iceberg table, the service account also needs BigLake connection permissions. Run bq show --format=prettyjson your_dataset.your_table to confirm the external table's connection is active.

Row count in target doesn't match SE16N

A mismatch typically means the filter scope differs between SAP and your sink query. Confirm the same RBUKRS, RYEAR (and POPER if applicable) filters are used in both SE16N and the count query against your sink. Also check for type conversion issues — NUMC fields stored as integers in the target may not match the string representation in SE16N's filter.
