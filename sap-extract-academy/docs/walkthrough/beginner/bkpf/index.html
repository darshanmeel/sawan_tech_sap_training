Extract BKPF Accounting Document Header (One Fiscal Year)

SAP-side preparation for beginner BKPF extraction. Covers fiscal year and company code partitioning, the released CDS view I_AccountingDocument, authorization setup, and SE16N row-count reconciliation.

Estimated time
50 minutes
Destinations
Generic
Extractors
ODP

SAP-side preflight

Before any tool touches your SAP system, you need four things confirmed on the SAP side. Skipping these steps is the most common cause of extraction failures, performance incidents, and licensing surprises.

  1. License — confirm your extraction method is covered.

    Runtime vs Full Use is the first decision. ODP-based extraction (via ODP) is permitted under Runtime licenses. SLT replication requires a Full Use license. Verify with your SAP licensing contact before you design anything — not the day before go-live.

    SAP Note 3255746 (Feb 2024): Use of ODP RFC function modules by third-party tools is no longer permitted. If your tool uses ODP via RFC (ADF SAP CDC, Qlik, Fivetran ODP, etc.), verify your contract terms. Compliant alternatives are ODP via OData or SLT replication. Full note at support.sap.com/notes/3255746 (SAP login required). [NEEDS_SAP_CITATION — verify current note version]

  2. Released CDS view — confirm the extraction-safe view exists.

    Every table covered in this Academy has a SAP-released CDS view annotated for extraction. Open SE80 and search for the view name before you build anything. Confirm the annotation @Analytics.dataExtract: true is present. If the view is absent or the annotation is missing, stop — raw table extraction is not the solution.

  3. Authorizations — grant the extraction user minimum permissions.

    Your extraction user needs three authorization objects. Use SU01 to create a System-type (non-dialog) user, then use PFCG to assign a role containing:

    • S_RFC — allows RFC calls (set RFC_TYPE=FUNC)
    • S_ODP_READ — allows reading from ODP contexts (set ODPSOURCE=ABAP_CDS for CDS-based extraction)
    • S_TABU_DIS — table display access for the relevant authorization group

    For SLT-based extraction, your Basis team will also need authorization in LTCO and LTRS. [NEEDS_SAP_CITATION — confirm exact auth objects for LTCO/LTRS]

  4. Row-count context — understand data volumes before you run.

    Use SE16N to count rows in the target table with your intended filter (company code, fiscal year, material type, etc.) before you start the extraction. This gives you:

    • A baseline to reconcile against after the extract.
    • A sense of whether your partition is safe to extract (< 100M rows is generally safe for ODP; larger partitions may need further splitting or SLT).
    • Early warning if a filter is misconfigured (count returns billions — rethink the partition).

These four checks take 15–30 minutes on a first walkthrough, and under 5 minutes once you know your system. They are the difference between a smooth extraction and a multi-day incident.

License check

ODP extraction is permitted under all license types. License trap: Runtime vs Full Use →

ODP (Operational Data Provisioning) is permitted under both Runtime and Full Use SAP licenses when you access data through published CDS views via the OData API.

SAP Note 3255746 (Feb 2024): Third-party tools that call ODP via RFC (ADF SAP CDC, Qlik, Fivetran ODP, Informatica ODP, etc.) are no longer permitted. Use ODP via OData, or switch to SLT. Verify with your licensing contact. Full note (SAP login required). [NEEDS_SAP_CITATION — confirm current note state]

SLT (SAP Landscape Transformation) requires a Full Use SAP license. It is not permitted under Runtime licenses. Verify in SLICENSE that "SLT Landscape Transformation" is listed as Active, Type = Full Use before you configure anything in LTCO.

License trap: Runtime vs Full Use →

Direct RFC extraction (e.g., RFC_READ_TABLE) is generally permitted for read-only access under Runtime and Full Use licenses, but is not recommended for production use — it bypasses authorization checks that released CDS views enforce and is subject to SAP Note 3255746 restrictions when wrapped by third-party ODP connectors. Overview

Your finance team needs accounting documents in the data lake for reconciliation and analytics. BKPF is large, so you start small: one company code, one fiscal year. This walkthrough covers the SAP-side preparation — confirming the CDS view, checking authorizations for financial data, and knowing how to verify the extraction with ODQMON and SE16N.

For fiscal year 2024 on a mid-sized system, expect 5M–30M document headers in one company code. That is a safe beginner partition for ODP-based extraction.

What you have established

You have the SAP side ready for BKPF extraction: released CDS view confirmed, financial authorization checked, ODQMON subscription verified, and a row-count reconciliation baseline recorded. Your tool team can now run the extraction against I_AccountingDocument with confidence that the SAP configuration is correct.

When 2025 arrives, you will repeat steps 02 and 05 with GJAHR=2025 — the rest stays the same.

Before you start

  • Completed VBAK or LFA1 beginner walkthrough
  • S/4HANA access with SE80, SE16N, and SU01 authorization
0 of 5 steps complete
  1. Step 1

    Identify the released CDS view I_AccountingDocument

    For BKPF, the released CDS view is I_AccountingDocument. It exposes accounting document headers through ODP with authorization enforcement. Confirm it exists in your system and carries the @Analytics.dataExtract: true annotation.

    Do this in SAP
    Open transaction SE80 (opens in new tab) and navigate to Repository Browser → I_AccountingDocument.
    Verify
    I_AccountingDocument appears in SE80 with @Analytics.dataExtract: true.
  2. Step 2

    Understand BKPF partitioning (BUKRS, GJAHR)

    BKPF is a large financial table. Its primary key includes MANDT, BUKRS (company code), GJAHR (fiscal year), and BELNR (document number). Use SE16N to count rows for one company code and one fiscal year. That partition count is your extraction target and your reconciliation baseline.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → BKPF → Filter BUKRS='1000' AND GJAHR=2024 → Count.
    Verify
    You see a row count for BUKRS='1000' and GJAHR=2024. Write it down.
  3. Step 3

    Confirm the extraction user has FI authorization group

    BKPF data is finance-sensitive. Verify the extraction user (created in the VBAK beginner walkthrough or equivalent) has S_TABU_DIS with authorization group FC (Finance) in addition to the S_RFC and S_ODP_READ already in place. If you are reusing EXTRACT_VBAK, check via PFCG whether the role covers finance tables.

    Do this in SAP
    Open transaction PFCG (opens in new tab) and navigate to Role → Display → Authorization Data → S_TABU_DIS.
    Verify
    S_TABU_DIS includes authorization group FC or the extraction user receives rows in a preview without authorization errors.
  4. Step 4

    Verify ODQMON subscription after first tool connection

    After your extraction tool makes its first connection attempt, check ODQMON. A subscription for I_AccountingDocument should appear under context ABAP_CDS.

    Do this in SAP
    Open transaction ODQMON (opens in new tab) and navigate to Subscriptions → Context: ABAP_CDS → I_AccountingDocument.
    Verify
    Subscription for I_AccountingDocument is active. No errors in ODQMON.
  5. Step 5

    Reconcile row count after extraction

    After the extraction completes, run the SE16N count again with the same BUKRS and GJAHR filter. Compare the result to the row count your landing zone reports.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → BKPF → Filter BUKRS='1000' AND GJAHR=2024 → Count.
    Verify
    Row count in landing zone matches SE16N count. Sample documents show posting dates in fiscal year 2024, company code 1000.

Troubleshooting

Filter condition returns zero rows in tool preview

In ODP, filter syntax is case-sensitive. BUKRS='1000' uses single quotes; GJAHR is numeric — in many extractors GJAHR=2024 (no quotes). Verify the exact syntax your tool expects.

Extraction times out despite BUKRS+GJAHR filter

Even one company code plus one fiscal year can hold 100M+ rows on large systems. If SE16N shows more than 100M rows for your partition, consider sub-partitioning by document type (BLART) or posting date month. Alternatively, use the Intermediate walkthrough which covers this.
