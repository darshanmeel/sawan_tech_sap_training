Extract VBAK Sales Document Header (Beginner)

SAP-side preparation for full-load extraction of the Sales Document Header. Covers the released CDS view I_SalesDocument, technical user setup with minimum authorizations, and post-extraction row-count reconciliation using SE16N.

Estimated time
45 minutes
Destinations
Generic
Extractors
ODP

Four things to confirm before any tool runs

The checks below, in order, are the difference between a smooth extraction and a multi-day incident. Each one is one sentence; expand for the how.

  1. 01 · Licence

    Confirm your extraction method is covered by your installed SAP licence before anyone designs anything.

    How to check

    Runtime licences cover ODP through OData. SLT replication requires a Full Use licence. Ask your SAP licensing contact; do not assume. See the dedicated article on Runtime vs Full Use, and note that SAP Note 3255746 prohibits third-party tools from using ODP via RFC — compliant paths are ODP over OData or SLT. [NEEDS_SAP_CITATION]

  2. 02 · Released CDS view

    Every table covered here has an extraction-safe released CDS view; confirm it exists before building.

    How to check

    Open the view in SE80 (or in ADT) and confirm the header annotation @Analytics.dataExtract: true is present. If it is missing, the view is not extractable — raw-table reads are not the fix.

  3. 03 · Authorisations

    A System-type technical user with the minimum authorisation objects for your chosen method — no named users, no wildcards.

    How to check

    Create a System user via SU01 and assign a role via PFCG. For ODP: S_RFC (RFC_TYPE=FUNC) and S_ODP_READ (ODPSOURCE=ABAP_CDS). For SLT your Basis team also needs authorisation in LTCO and LTRS. For RFC direct add S_TABU_DIS scoped to the table's authorisation group — never wildcard it.

  4. 04 · Row-count baseline

    Use SE16N to measure the intended partition before extraction — you cannot reconcile without this number.

    How to check

    Open SE16N, apply the same filter your extraction will use, and record the count. Under 100M rows is usually safe for a single ODP partition; larger volumes need further splitting or SLT. A count in the billions means the filter is misconfigured — rethink the partition before running anything.

License check

Works under both Full Use and Runtime SAP licenses. ODP extraction through the OData API is permitted under all license types. License trap: Runtime vs Full Use →

ODP (Operational Data Provisioning) is permitted under both Runtime and Full Use SAP licenses when you access data through published CDS views via the OData API.

SAP Note 3255746 (Feb 2024): Third-party tools that call ODP via RFC (ADF SAP CDC, Qlik, Fivetran ODP, Informatica ODP, etc.) are no longer permitted. Use ODP via OData, or switch to SLT. Verify with your licensing contact. Full note (SAP login required). [NEEDS_SAP_CITATION — confirm current note state]

SLT (SAP Landscape Transformation) requires a Full Use SAP license. It is not permitted under Runtime licenses. Verify in SLICENSE that "SLT Landscape Transformation" is listed as Active, Type = Full Use before you configure anything in LTCO.

License trap: Runtime vs Full Use →

Direct RFC extraction (e.g., RFC_READ_TABLE) is generally permitted for read-only access under Runtime and Full Use licenses, but is not recommended for production use — it bypasses authorization checks that released CDS views enforce and is subject to SAP Note 3255746 restrictions when wrapped by third-party ODP connectors. Overview

Your finance analytics project needs sales order data in the data lake. Specifically, the header record for every sales order — orders, quotations, contracts. This walkthrough covers the SAP-side work from confirming the released CDS view to verifying the extraction succeeded.

Your tool team handles pipeline configuration (linked services, copy activities, Parquet sinks). This walkthrough is about the SAP side: what to check, which transactions to open, what authorizations to set, and how to know the extraction is correct.

If you complete this in one sitting, budget about 45 minutes. If this is your first time navigating SAP transactions, add 30 minutes.

What you have established

You have confirmed that the SAP side is correctly configured for VBAK extraction: the released CDS view is present, a minimal-permission extraction user is in place, ODP is registering subscriptions, and you have a row-count baseline for reconciliation.

The patterns you have learned — released CDS views, System-type RFC users, S_ODP_READ authorization, ODQMON monitoring, SE16N reconciliation — are the foundation for every S/4HANA extraction. When you move to the intermediate walkthrough, you will add delta processing and a Z-field.

Before you start

  • Access to an S/4HANA on-premise system with SE80 and SE16N authorization
  • Ability to create or request a technical user via SU01
  • Awareness of your SAP license type
0 of 7 steps complete
  1. Step 1

    Confirm the released CDS view I_SalesDocument

    SAP ships released CDS views for standard tables. For VBAK, the view is I_SalesDocument. It carries authorization, delta support, and the @Analytics.dataExtract: true annotation that signals ODP readiness. Using this view is the SAP-recommended approach — extracting raw VBAK bypasses authorization and has no delta support.

    Do this in SAP
    Open transaction SE80 (opens in new tab) and navigate to Repository Browser → Search → I_SalesDocument.
    Verify
    I_SalesDocument opens in SE80 and the annotation @Analytics.dataExtract: true is visible in the header.
    Why this matters

    Extracting from a released CDS view respects authorization and provides built-in delta support. Reading VBAK directly skips both. Released views are also more stable across upgrades than raw table layouts.

  2. Step 2

    Check row count in SE16N (establish a reconciliation baseline)

    Use SE16N to count rows in VBAK without filters (or with the filter that matches your intended extraction scope). Write down the number. This is your target: after the extraction, the row count in your landing zone must match this number.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → VBAK → Count.
    Verify
    You have a row count for VBAK. Typical range is 1M–50M rows depending on system size and history retained.
  3. Step 3

    Create a technical RFC user with minimum permissions

    Create a System-type (non-dialog) user via SU01. System users cannot log on interactively and do not consume a dialog work process. Never reuse a named user account for automated extraction — it creates audit, security, and performance problems.

    Do this in SAP
    Open transaction SU01 (opens in new tab) and navigate to User → Create → User type: System.
    Verify
    User EXTRACT_VBAK exists in SU01 with User Type = System. The user is locked for interactive logon.
  4. Step 4

    Assign authorizations via PFCG

    In PFCG, create a single custom role. Add three authorization objects: S_RFC (RFC_TYPE=FUNC), S_ODP_READ (ODPSOURCE=ABAP_CDS), and S_TABU_DIS (authorization group SD). Generate the profile and assign the role to EXTRACT_VBAK.

    Do this in SAP
    Open transaction PFCG (opens in new tab) and navigate to Role → Create → Authorizations → Add Object.
    Verify
    In SU01, EXTRACT_VBAK shows one role assigned. In PFCG, the role authorization data contains S_ODP_READ with ODPSOURCE=ABAP_CDS.
  5. Step 5

    Verify the RFC destination is reachable (SM59)

    Your extraction tool connects to SAP over an RFC destination. Ask your Basis team to check SM59 for the destination your tool will use. The test connection must pass before any extraction attempt.

    Do this in SAP
    Open transaction SM59 (opens in new tab) and navigate to RFC Connections → ABAP Connections or TCP/IP → Test Connection.
    Verify
    Test Connection returns green for the RFC destination your tool uses.
  6. Step 6

    Check ODQMON after the tool's first run

    After your extraction tool makes its first connection, open ODQMON. A subscription for I_SalesDocument should appear under context ABAP_CDS. No subscription means the tool did not register with ODP — this usually means it is using an unsupported path or the authorization is incorrect.

    Do this in SAP
    Open transaction ODQMON (opens in new tab) and navigate to Subscriptions → Context: ABAP_CDS → I_SalesDocument.
    Verify
    ODQMON shows an active subscription for I_SalesDocument. No error status.
    Why this matters

    ODQMON is your diagnostic console. If the extraction looks successful in your tool but the data count is wrong, ODQMON will show you why.

  7. Step 7

    Reconcile row counts after extraction

    Return to SE16N and count VBAK rows with the same filters you used in step 02. Compare to the row count your landing zone reports. The counts should match or differ only by in-flight postings that arrived during the extraction window.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → VBAK → Count.
    Verify
    Row count in landing zone matches SE16N count within a tolerable margin.
    Why this matters

    Row-count reconciliation is the single most important verification in any extraction. Never hand data to the analytics team without it.

  1. Step 1

    Basis: confirm Full Use licence in SLICENSE

    SLT replication is only permitted under a Full Use SAP licence. Ask your Basis team to open SLICENSE and confirm the installed licence type. A Runtime licence is not enough — replicating through SLT under Runtime is a contract breach. If the licence is Runtime, stop this walkthrough and use the ODP variant instead.

    Do this in SAP
    Open transaction SLICENSE (opens in new tab) and navigate to System → Licence → Installed Licences.
    Verify
    Installed licence row shows a product line that explicitly permits SLT replication (typically SAP S/4HANA Full Use or equivalent). Screenshot this and attach to your design doc.
    Why this matters

    Licence verification is the single most common thing SLT projects skip — and the single most expensive thing to discover at go-live. Always front-load this check.

  2. Step 2

    Developer: inspect VBAK and I_SalesDocument in ADT

    Open VBAK and the released CDS view I_SalesDocument in ABAP Development Tools (ADT) in Eclipse. You need to see two things. First, the raw table columns of VBAK (MANDT, VBELN, ERDAT, AUART, VKORG, VTWEG, KUNNR …) — SLT replicates the raw table, not the CDS view, so the raw layout is what lands in the target. Second, the released CDS view so you understand what downstream analytics would normally consume; SLT output will need a comparable semantic layer on top.

    Do this in SAP
    Open transaction ADT (Eclipse) (opens in new tab) and navigate to Project Explorer → System → Dictionary → Database Tables → VBAK.
    Verify
    In ADT you can see the VBAK table definition with its 100+ raw columns and its delivery class A (application table). You can open I_SalesDocument source and read the SELECT projection. Note that the column names differ (raw VBAK is technical, CDS is semantic) — your downstream layer must map between them.
    Why this matters

    SLT is raw-table replication. If your team is used to CDS semantic names (SalesDocument, CreationDate), the SLT output (VBELN, ERDAT) will surprise them. Agreeing on the naming mapping now avoids a rename project later.

  3. Step 3

    Basis: configure the SLT replication object in LTRC

    Your Basis team opens LTRC on the SLT system and creates a replication configuration pointing at the source S/4HANA and the target (RFC destination for a database, or a logging-table-based sink). Add table VBAK with replication mode "Load + Replication" for full load plus ongoing delta.

    Do this in SAP
    Open transaction LTRC (opens in new tab) and navigate to Configurations → Create → Source: S/4HANA system · Target: ... · Replication.
    Verify
    LTRC shows a configuration row for VBAK with status Configured. The source and target RFC destinations both test green.
    Why this matters

    LTRC is the SLT control plane. Everything else — triggers, parallelism, delta — is configured from here.

  4. Step 4

    Basis: set parallel reader count in LTRS

    For a small table like VBAK, one or two parallel readers is enough. Your Basis team opens LTRS and sets the Reader/Job count for VBAK. Keep initial load jobs ≤ 2 for a beginner setup; more readers generate more dialog-process pressure and are rarely needed for a header-only table.

    Do this in SAP
    Open transaction LTRS (opens in new tab) and navigate to Specific Settings → Performance Options → Number of Parallel Jobs.
    Verify
    LTRS shows VBAK with a reader count of 1 or 2 and a calculation type appropriate for the key (most commonly KEY_BASED for VBAK).
  5. Step 5

    Basis: start replication and monitor in LTCO

    Trigger Load + Replication from LTRC and watch progress in LTCO. LTCO shows per-table load status, replication lag, and job health. For VBAK the initial load should complete in minutes; once it is in Replication state, inserts and updates on the source appear at the target within seconds.

    Do this in SAP
    Open transaction LTCO (opens in new tab) and navigate to Control → Configurations → VBAK.
    Verify
    LTCO shows VBAK with status Replication (green). The row count on the target matches the source, and a new order posted on the source appears in the target within a minute.
    Why this matters

    LTCO is where you discover problems. A VBAK stuck in Initial Load means the target is slower than the source or a trigger failed — either way you need Basis before Analytics gets curious.

  6. Step 6

    Basis + developer: confirm no ODP subscription is needed

    SLT does not use ODP — it writes straight into the target through triggers and the SLT read module. So there is nothing to check in ODQMON for this path. If your design mentions ODQMON or ABAP_CDS context, it is accidentally the ODP design — reopen the method picker and switch to ODP.

    Verify
    Your design doc says SLT everywhere and does not mention ODP/ODQMON/ABAP_CDS. If it does, it is mixing two methods and one of them is wrong.
  7. Step 7

    Row-count reconcile against source

    Compare VBAK row count between source and target. Use SE16N on the source with the same filter, compare against SELECT COUNT(*) on the target. A delta of zero is the goal; small positive deltas on an active source are normal and explained by the handful of orders posted during the counting window.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → VBAK → Count → (record the number).
    Verify
    Source and target counts match exactly, or differ by a small number you can explain (count of orders posted during the reconciliation window).
  1. Step 1

    Developer: inspect VBAK structure in SE11 or ADT

    RFC direct reads the raw table. Open VBAK in SE11 (classic) or in ADT (modern). Note the key fields (MANDT, VBELN) and the authorization group — for sales documents this is usually V or VB. You will need the correct authorization group in PFCG later.

    Do this in SAP
    Open transaction SE11 (opens in new tab) and navigate to Database Table → VBAK → Display → Delivery and Maintenance.
    Verify
    You can read the delivery class (A), the authorization group (V or VB), and the full key list for VBAK.
    Why this matters

    RFC direct extraction bypasses released CDS views, so you inherit no authorization, no annotation, and no built-in delta. Knowing the raw structure up front is the only thing keeping the extraction honest.

  2. Step 2

    Measure row count in SE16N (reconciliation baseline)

    Count rows in VBAK with the filter you intend to extract. Record the number. This is the count the landing zone must match after the extract completes. Without this number, reconciliation is impossible.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → VBAK → Enter filter → Count.
    Verify
    You have a number in hand (or in the design doc) for VBAK under your chosen filter.
  3. Step 3

    Basis: create technical user via SU01

    Ask Basis to create a System-type user (not Dialog) in SU01. System users cannot log on interactively. Without a dedicated system user, any extraction that runs on behalf of a named person becomes an audit finding.

    Do this in SAP
    Open transaction SU01 (opens in new tab) and navigate to User → Create → User type: System.
    Verify
    EXTRACT_VBAK exists with type System and is locked for interactive logon.
  4. Step 4

    Basis: role with S_RFC + S_TABU_DIS (authgroup V) in PFCG

    In PFCG, create a single-role profile. Add S_RFC with RFC_TYPE=FUNC, and S_TABU_DIS with DICBERCLS=V (the Sales authorization group covering VBAK). Do not grant S_TABU_DIS with * — that is a finding.

    Do this in SAP
    Open transaction PFCG (opens in new tab) and navigate to Role → Create → Authorizations → Add Object.
    Verify
    In SU01 the extraction user shows the role. In PFCG the role contains exactly the two objects above, scoped to the minimum needed.
  5. Step 5

    Basis: test RFC destination in SM59

    The tool calls SAP over an RFC destination defined in SM59. Ask Basis to Test Connection and Test Authorization against the destination the tool uses. Both must pass before a single row is extracted.

    Do this in SAP
    Open transaction SM59 (opens in new tab) and navigate to RFC Connections → ABAP Connections or TCP/IP → Test Connection → Test Authorization.
    Verify
    Test Connection returns green. Test Authorization returns green for the extraction user.
  6. Step 6

    Basis: watch SM50 during the first extraction

    During the first run, ask Basis to keep SM50 open on the application server hosting the RFC. RFC direct consumes dialog work processes for the lifetime of the call. If the extraction dominates the work-process pool, finance users will notice.

    Do this in SAP
    Open transaction SM50 (opens in new tab) and navigate to Processes → Filter by User: EXTRACT_VBAK.
    Verify
    At any moment during the extract, total dialog-process utilisation on the app server stays below 80%. If it spikes higher, throttle the extraction to one concurrent call.
  7. Step 7

    Row-count reconcile after extraction

    Return to SE16N with the same filter used in step 02. Compare to the row count in the landing zone. Matching or in-flight-only delta = success. Anything else, investigate before trusting the data.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → VBAK → same filter → Count.
    Verify
    Landing-zone count matches SE16N count within a tolerable margin (e.g. new orders posted during the extraction window).
  1. Step 1

    What VBAK is and why extract it

    VBAK is the sales document header. One row per order, quotation, or contract. The released CDS view for this table is I_SalesDocument — it is authorisation-aware, delta-capable, and ODP-ready. For line items, the corresponding table is VBAP.

    Verify
    You can explain in one sentence what lives in VBAK and why analytics teams want it.
  2. Step 2

    Methods at a glance (ODP / SLT / RFC direct)

    ODP is the recommended path — runs under both Runtime and Full Use licences, reads through the released CDS view, and plays nicely with ODQMON for monitoring. SLT is raw-table replication with live delta, requires a Full Use licence, and replicates to the target at the database level. RFC direct is a last-resort path with no delta and requires careful authorization scoping.

    Verify
    You can name the one reason an organisation would pick each method.

Troubleshooting

Test Connection in SM59 fails

Check that network rules allow the tool's server to reach SAP's message server port (usually 3600+systemNumber) and application server host. Engage your Basis team — this is a network or SAP host configuration issue, not an extraction configuration issue.

SAP_ODP_AUTHORIZATION_MISSING error

The EXTRACT_VBAK user lacks S_ODP_READ for context ABAP_CDS. Return to PFCG and add this authorization with ODPSOURCE=ABAP_CDS. Regenerate the profile and test again.

ODQMON shows no subscription after tool runs

The tool may be calling ODP via RFC rather than OData. RFC-based ODP is restricted per SAP Note 3255746. Check the tool's connector configuration and switch to OData mode.
