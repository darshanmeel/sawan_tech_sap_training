beginner ACDOCA

Extract ACDOCA Universal Journal (One Fiscal Year, Generic Target)

Beginner extraction of one company code and one fiscal year from the Universal Journal. Introduces the mandatory partitioning strategy for ACDOCA — the heaviest table in S/4HANA — and writes output to a generic landing zone.

Estimated time
60 minutes
Destinations
ADLS, Generic
Extractors
ADF

License check

ODP-based extraction of ACDOCA works under Runtime License. SLT replication of ACDOCA requires Full Use License. This walkthrough uses ODP (CDC connector) and is Runtime-safe.

Overview

ACDOCA is the Universal Journal — every financial posting in S/4HANA ends up here. Unlike VBAK or LFA1, ACDOCA has no safe beginner path that ignores partitioning. The table routinely exceeds a billion rows in production systems. This walkthrough gives you the minimum viable extraction: one company code, one fiscal year, written to a generic landing zone.

Completing this walkthrough means you have confirmed the partition strategy works and have a Parquet file you can query. Intermediate and expert walkthroughs build on this by adding parallelism, delta extraction, and multi-year full loads.

Before you start

  • Completed at least one beginner walkthrough (VBAK or BKPF recommended)
  • S/4HANA on-premise access with SE80 and SE16N authorization
  • Technical RFC user with S_ODP_READ authorization for ABAP_CDS context
  • A target landing zone (ADLS, local filesystem, or any supported sink)
0 of 5 steps complete
  1. Step 1

    Confirm the released CDS view I_JournalEntryItem

    Never extract raw ACDOCA directly. SAP ships I_JournalEntryItem as the released CDS view for the Universal Journal. It enforces authorization, applies currency conversion, and exposes the table through ODP without triggering a full table scan.

    Do this in SAP
    Open transaction SE80 (opens in new tab) and navigate to Repository Browser → Search → I_JournalEntryItem.
    Verify
    I_JournalEntryItem opens in SE80 and the header annotation @Analytics.dataExtract: true is present.
    Why this matters

    A raw SELECT * on ACDOCA exhausts SAP dialog work process memory within seconds on any production system. I_JournalEntryItem is the only extraction-safe path.

  2. Step 2

    Understand ACDOCA partitioning (RBUKRS, RYEAR, POPER)

    ACDOCA primary key fields are MANDT, RBUKRS (company code), RYEAR (reporting year), POPER (posting period), BELNR (document number), and DOCLN (line number). For a beginner extraction, always fix to one company code and one reporting year. This gives a manageable partition that completes in minutes rather than hours.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → ACDOCA → Filter RBUKRS='1000' AND RYEAR=2024.
    Verify
    You see a row count for RBUKRS='1000' and RYEAR=2024. This is the expected output row count for your first partition. Typical values are 5M-50M rows per company code per year.
    Why this matters

    ACDOCA without partitioning crashes SAP. Partitioning by RBUKRS+RYEAR is the minimum viable approach and is the pattern you will scale to parallel jobs in intermediate and expert walkthroughs.

  3. Step 3

    Create a source dataset pointing to I_JournalEntryItem

    In your extraction tool (ADF or equivalent), create a new source dataset of type SAP ODP Resource pointing to the I_JournalEntryItem context. Add filter parameters for RBUKRS and RYEAR so you can fix them to one value for this walkthrough.

    Verify
    Preview returns rows and the column list includes CompanyCode, FiscalYear, PostingDate, AmountInCompanyCodeCurrency, and CurrencyCode.
  4. Step 4

    Create the target folder structure

    Create a folder path in your landing zone that reflects the partition. A clear naming convention is essential when you scale to multiple company codes and years.

    Verify
    The target folder exists and is empty before you run the pipeline.
  5. Step 5

    Configure the copy pipeline and run

    Create a Copy activity with the ACDOCA source and your target sink. Set the sink format to Parquet with Snappy compression — ACDOCA rows are wide and Parquet column pruning will significantly reduce storage. Run the pipeline and monitor progress.

    Verify
    Pipeline completes with status Succeeded. The output folder contains one or more .parquet files. Row count in the pipeline monitor matches your SE16N estimate from step 02.
    Why this matters

    parallelCopies: 1 keeps the extraction single-threaded, which is safest for a first run on ACDOCA. Intermediate walkthrough covers parallel execution.

You've built:

Congratulations! You've completed the beginner walkthrough for ACDOCA. You now have 5 steps of experience extracting this table, and you understand the key considerations for the ACDOCA use case.
