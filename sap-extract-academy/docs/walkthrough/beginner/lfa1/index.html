License check

Works under all SAP license types. ODP extraction via OData is permitted for all licenses. License trap: Runtime vs Full Use →

ODP (Operational Data Provisioning) is permitted under both Runtime and Full Use SAP licenses when you access data through published CDS views via the OData API.

SAP Note 3255746 (Feb 2024): Third-party tools that call ODP via RFC (ADF SAP CDC, Qlik, Fivetran ODP, Informatica ODP, etc.) are no longer permitted. Use ODP via OData, or switch to SLT. Verify with your licensing contact. Full note (SAP login required). [NEEDS_SAP_CITATION — confirm current note state]

SLT (SAP Landscape Transformation) requires a Full Use SAP license. It is not permitted under Runtime licenses. Verify in SLICENSE that "SLT Landscape Transformation" is listed as Active, Type = Full Use before you configure anything in LTCO.

License trap: Runtime vs Full Use →

Direct RFC extraction (e.g., RFC_READ_TABLE) is generally permitted for read-only access under Runtime and Full Use licenses, but is not recommended for production use — it bypasses authorization checks that released CDS views enforce and is subject to SAP Note 3255746 restrictions when wrapped by third-party ODP connectors. SAP-side preflight

Before any tool touches your SAP system, you need four things confirmed on the SAP side. Skipping these steps is the most common cause of extraction failures, performance incidents, and licensing surprises.

  1. License — confirm your extraction method is covered.

    Runtime vs Full Use is the first decision. ODP-based extraction (via ODP) is permitted under Runtime licenses. SLT replication requires a Full Use license. Verify with your SAP licensing contact before you design anything — not the day before go-live.

    SAP Note 3255746 (Feb 2024): Use of ODP RFC function modules by third-party tools is no longer permitted. If your tool uses ODP via RFC (ADF SAP CDC, Qlik, Fivetran ODP, etc.), verify your contract terms. Compliant alternatives are ODP via OData or SLT replication. Full note at support.sap.com/notes/3255746 (SAP login required). [NEEDS_SAP_CITATION — verify current note version]

  2. Released CDS view — confirm the extraction-safe view exists.

    Every table covered in this Academy has a SAP-released CDS view annotated for extraction. Open SE80 and search for the view name before you build anything. Confirm the annotation @Analytics.dataExtract: true is present. If the view is absent or the annotation is missing, stop — raw table extraction is not the solution.

  3. Authorizations — grant the extraction user minimum permissions.

    Your extraction user needs three authorization objects. Use SU01 to create a System-type (non-dialog) user, then use PFCG to assign a role containing:

    • S_RFC — allows RFC calls (set RFC_TYPE=FUNC)
    • S_ODP_READ — allows reading from ODP contexts (set ODPSOURCE=ABAP_CDS for CDS-based extraction)
    • S_TABU_DIS — table display access for the relevant authorization group

    For SLT-based extraction, your Basis team will also need authorization in LTCO and LTRS. [NEEDS_SAP_CITATION — confirm exact auth objects for LTCO/LTRS]

  4. Row-count context — understand data volumes before you run.

    Use SE16N to count rows in the target table with your intended filter (company code, fiscal year, material type, etc.) before you start the extraction. This gives you:

    • A baseline to reconcile against after the extract.
    • A sense of whether your partition is safe to extract (< 100M rows is generally safe for ODP; larger partitions may need further splitting or SLT).
    • Early warning if a filter is misconfigured (count returns billions — rethink the partition).

These four checks take 15–30 minutes on a first walkthrough, and under 5 minutes once you know your system. They are the difference between a smooth extraction and a multi-day incident.

Overview

Your procurement analytics team needs a periodic feed of all active vendors. LFA1 (vendor master general data) is small, stable, and does not change hourly — a full-load extraction on a weekly or daily schedule is the correct pattern.

This walkthrough covers the SAP-side preparation: confirming the released CDS view, checking authorization for vendor master data, and verifying the extraction with ODQMON and SE16N. Your tool team handles the pipeline and scheduling.

Estimated time: 40 minutes, faster if you have already completed the VBAK beginner walkthrough and your extraction user is already configured.

What you have established

You have confirmed the SAP side is correctly configured for LFA1 extraction. The released CDS view is present, the extraction user has vendor master authorization, and you have a row-count baseline. Your tool team can now configure the extraction pipeline against I_Supplier with confidence.

The patterns here — CDS view, System-type user, ODQMON subscription check, SE16N reconciliation — are identical for every table. When you move to the intermediate walkthrough, you will add delta processing.

Before you start

  • Access to an S/4HANA on-premise system with SE80 and SE16N authorization
  • Ability to create or request a technical user via SU01

Step-by-step

0 of 5 steps complete
  1. Step 1

    Verify the released CDS view I_Supplier

    Vendor master general data is exposed via the I_Supplier CDS view. Confirm it is present in your system and annotated for extraction.

    Do this in SAP
    Open transaction SE80 (opens in new tab) and navigate to Repository Browser → Search → I_Supplier.
    Verify
    I_Supplier appears in SE80 with @Analytics.dataExtract: true.
  2. Step 2

    Count rows in LFA1 (SE16N baseline)

    Use SE16N to count all rows in LFA1 without filters. LFA1 is typically small (thousands to low millions of vendors). Note the count for reconciliation.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → LFA1 → Count (no filter).
    Verify
    You have a row count for LFA1. Typical range: 50K–2M vendors.
  3. Step 3

    Confirm the extraction user has vendor master authorization

    Vendor master sits in authorization group MM (Materials Management) for table display. Confirm the extraction user has S_TABU_DIS for group MM in addition to S_RFC and S_ODP_READ. Use PFCG to check or update.

    Do this in SAP
    Open transaction PFCG (opens in new tab) and navigate to Role → Display → Authorization Data.
    Verify
    Role covers S_TABU_DIS for authorization group MM (or a broader group that includes vendor master tables).
  4. Step 4

    Check ODQMON after the first tool run

    After your extraction tool makes its first connection, verify ODQMON shows an active subscription for I_Supplier.

    Do this in SAP
    Open transaction ODQMON (opens in new tab) and navigate to Subscriptions → Context: ABAP_CDS → I_Supplier.
    Verify
    Subscription for I_Supplier is active. No errors.
  5. Step 5

    Reconcile vendor count after extraction

    Return to SE16N and count LFA1 again. The row count in your landing zone must match. LFA1 is stable enough that any significant discrepancy indicates a configuration problem, not in-flight updates.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → LFA1 → Count.
    Verify
    Landing zone row count matches SE16N count. Sample columns LIFNR, NAME1, LAND1 have expected values.

Troubleshooting

ODP_NO_DATA_AVAILABLE error in tool

Verify LFA1 has rows via SE16N. Then check if I_Supplier applies a hidden filter — open the view in SE80 and look for WHERE clause conditions that might restrict active vendor status. If the view filters to only active vendors, the extracted count will be less than raw LFA1.

ODQMON shows no subscription

The tool may be using RFC-based ODP, which is restricted per SAP Note 3255746. Confirm the tool is using OData mode and re-run.
