Overview

Scenario

Your finance warehouse needs three years of accounting documents. You will extract in parallel by fiscal year and company code to minimize extraction time, while keeping SAP work process utilization safe for concurrent posting activity.

What you have confirmed

You have a per-partition row-count baseline, know how to monitor SAP work process load during parallel extractions, and understand how to use ODQMON to track subscription health across multiple concurrent extractions. Enterprise-scale extraction (billions of rows, SLT) is covered in the Expert walkthrough.

Before you start

License check

All licenses. ODP via OData API. License trap: Runtime vs Full Use →

ODP (Operational Data Provisioning) is permitted under both Runtime and Full Use SAP licenses when you access data through published CDS views via the OData API.

SAP Note 3255746 (Feb 2024): Third-party tools that call ODP via RFC (ADF SAP CDC, Qlik, Fivetran ODP, Informatica ODP, etc.) are no longer permitted. Use ODP via OData, or switch to SLT. Verify with your licensing contact. Full note (SAP login required).

SLT (SAP Landscape Transformation) requires a Full Use SAP license. It is not permitted under Runtime licenses. Verify in SLICENSE that "SLT Landscape Transformation" is listed as Active, Type = Full Use before you configure anything in LTCO.

License trap: Runtime vs Full Use →

Direct RFC extraction (e.g., RFC_READ_TABLE) is generally permitted for read-only access under Runtime and Full Use licenses, but is not recommended for production use — it bypasses authorization checks that released CDS views enforce and is subject to SAP Note 3255746 restrictions when wrapped by third-party ODP connectors.

SAP prerequisites

These steps are one-time per SAP system (not repeated per table). Once your RFC user, authorizations, and RFC destination are set up, every table extraction reuses the same connection. The checks below, in order, are the difference between a smooth extraction and a multi-day incident.

  1. 01 · Licence

    Confirm your extraction method is covered by your installed SAP licence before anyone designs anything.

    How to check

    Runtime licences cover ODP through OData. SLT replication requires a Full Use licence. Ask your SAP licensing contact; do not assume. See the dedicated article on Runtime vs Full Use, and note that SAP Note 3255746 prohibits third-party tools from using ODP via RFC — compliant paths are ODP over OData or SLT.

  2. 02 · Released CDS view

    Every table covered here has an extraction-safe released CDS view; confirm it exists before building.

    How to check

    Open the view in SE80 (or in ADT) and confirm the header annotation @Analytics.dataExtract: true is present. If it is missing, the view is not extractable — raw-table reads are not the fix.

  3. 03 · Authorisations

    A System-type technical user with the minimum authorisation objects for your chosen method — no named users, no wildcards.

    How to check

    Create a System user via SU01 and assign a role via PFCG. For ODP: S_RFC (RFC_TYPE=FUNC) and S_ODP_READ (ODPSOURCE=ABAP_CDS). For SLT your Basis team also needs authorisation in LTCO and LTRS. For RFC direct add S_TABU_DIS scoped to the table's authorisation group — never wildcard it.

  4. 04 · Row-count baseline

    Use SE16N to measure the intended partition before extraction — you cannot reconcile without this number.

    How to check

    Open SE16N, apply the same filter your extraction will use, and record the count. Under 100M rows is usually safe for a single ODP partition; larger volumes need further splitting or SLT. A count in the billions means the filter is misconfigured — rethink the partition before running anything.

Also confirm

  • Technical extraction user (System type) created in SU01 with S_RFC and S_ODP_READ authorizations
  • SAP S/4HANA access with SE16N and SE80 authorization

SAP side

Steps to complete in your SAP system. Use the Method picker above to switch between ODP, SLT, and RFC approaches.

0 of 5 steps complete
  1. Step 1

    Count rows per partition in SE16N

    For each fiscal year and company code combination you plan to extract, run a count in SE16N with BUKRS and GJAHR filters. Record each count. This gives you a per-partition reconciliation baseline and helps you estimate total extraction time.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → BKPF → Filter BUKRS='1000' AND GJAHR=2022 → Count; repeat for 2023, 2024.
    Verify
    You have row counts for each (BUKRS, GJAHR) combination. Total row count across all partitions is documented.
  2. Step 2

    Confirm I_AccountingDocument CDS view is available for all target years

    The released CDS view I_AccountingDocument exposes BKPF through ODP. Verify it is active in SE80. Also confirm that your extraction does not need data from before the system cut-over date — for systems migrated from ECC to S/4HANA, historical data may not be in BKPF (it may be in BKPF_OLD or have been migrated). [NEEDS_SAP_CITATION — confirm BKPF_OLD behavior for migrated systems]

    Do this in SAP
    Open transaction SE80 (opens in new tab) and navigate to Repository Browser → I_AccountingDocument.
    Verify
    I_AccountingDocument is active. Ask Basis team whether pre-migration documents are in scope and where they are stored.
  3. Step 3

    Monitor SAP system load during multi-partition extraction (SM50)

    When running multiple partition extractions in parallel, your extraction tool opens multiple RFC connections simultaneously. Each connection consumes a work process. Monitor SM50 to confirm work process utilization stays below 80%. If it climbs higher, the finance team (who are posting documents) will be affected.

    Do this in SAP
    Open transaction SM50 (opens in new tab) and navigate to Work Process Overview.
    Verify
    During extraction, SM50 shows work processes for your extraction user (DIA or BTC type) with CPU and status. Total work process utilization stays below 80%.
    Why this matters

    If your extraction saturates all available work processes, finance users cannot post documents. Coordinate extraction windows with the finance team — run multi-partition extractions outside posting hours.

  4. Step 4

    Check ODQMON for multiple active subscriptions

    If your tool runs multiple partitions in parallel via multiple ODP subscriptions, ODQMON shows all active subscriptions. Monitor that each subscription is progressing and none is stuck in an error or fetching state. A stuck subscription consumes resources and blocks that partition.

    Do this in SAP
    Open transaction ODQMON (opens in new tab) and navigate to Subscriptions → Context: ABAP_CDS → Filter by I_AccountingDocument.
    Verify
    All subscriptions for I_AccountingDocument are in active (not error) state. Queue depths are decreasing over time.
  5. Step 5

    Reconcile totals per partition after extraction

    For each (BUKRS, GJAHR) partition, re-run the SE16N count you recorded in step 01 and compare to the row count in your target. All partitions must reconcile before you sign off on the extraction.

    Do this in SAP
    Open transaction SE16N (opens in new tab) .
    Verify
    Row counts in target match SE16N for each (BUKRS, GJAHR) partition. Document any acceptable variance for in-flight postings.
Troubleshooting
SM50 shows work process saturation during extraction

Reduce the number of parallel extraction connections. Limit to 3–4 concurrent partitions maximum. Queue remaining partitions to run sequentially after the first batch completes. Run extractions outside peak posting hours.

One partition fails while others succeed

BKPF for a specific fiscal year may have unusual volume (e.g., a year-end closing run generated more documents than normal). Re-run the failed partition with a smaller sub-partition (e.g., add a BLART filter for document types). Check ODQMON for the specific subscription error message.

+ Handling custom Z columns

ODP — Extending a CDS view with ABAP Developer

Use EXTEND VIEW ENTITY in ABAP Development Tools (Eclipse/ADT) to add Z columns without modifying the original view. The base view must carry the header annotation @AbapCatalog.viewEnhancementCategory: [#PROJECTION_LIST] — if the owner restricted data sources, only tables listed under @AbapCatalog.extensibility.dataSources can be joined.

  1. Open ADT (Eclipse). In the Project Explorer, locate the data definition of the CDS entity you want to extend.
  2. Right-click the data definition → New Data Definition. Choose your package, enter a name (e.g., Z_EXT_MY_VIEW), and add a description.
  3. Click Next, select your transport request, then choose the Extend View Entity template.
  4. Define your custom Z fields in the element list — pull from the underlying table alias or define a new association.
  5. Activate the view extension (Ctrl + F3). The base view now includes your custom elements.
extend view entity C_BaseStandardView with Z_Ext_My_View
{
  BaseTable.zz_custom_field    as ZZCustomField,
  _NewZTableAssoc
}

Because the ODP framework generates a TransientProvider at runtime from CDS metadata, Z columns added via EXTEND VIEW ENTITY automatically appear in the ODP delta queue. Refresh the schema in your tool (ADF, Databricks) to see new fields. Test locally with ABAP program RODPS_REPL_TEST.

SLT — Adding Z columns via LTRS Rule Assignment

In SLT you add and populate Z columns through Advanced Replication Settings in transaction LTRS.

  1. In your SLT system, run transaction LTRS.
  2. Under Table Settings, you can define the target table structure. To add a Z column that doesn't exist in the source, append it to the target structure here.
  3. Go to the Rule Assignment section → right-click → Add Table to add your replication table.
  4. Click the Create icon and choose either a Field-Related Rule (to populate a specific field) or an Event-Related Rule (to manipulate the whole record).
  5. Write ABAP in the Line of Code section to populate your Z column. For event-related rules use <wa_s_table>-columnname. For field-related rules use the E_ prefix: E_columnname.

RFC — Exposing Z columns via SE11 Append Structure

When extracting via RFC, Z columns added to a SAP transparent table through an Append Structure in transaction SE11 are immediately available — the RFC functions query the ABAP data dictionary dynamically.

  1. In SE11, open the transparent table, go to Extras → Append Structure, and add your Z field. Activate — the underlying database table is updated automatically.
  2. Standard RFC functions (RFC_READ_TABLE, /SAPDS/RFC_READ_TABLE2) will return the column dynamically.
  3. Explicitly request the new Z column in the FIELDS parameter of your RFC call — it is not returned unless listed there.
  4. If using a custom Function Module, update the Export table structure in the FM to include the new columns — it does not auto-discover.

Custom (Python / pyrfc) — multi-partition extraction

  1. Step 1 Extract BKPF for multiple fiscal years in parallel

    Use pyrfc to open multiple simultaneous connections (one per BUKRS+GJAHR partition). Each connection runs a full ODP extraction. Monitor SAP work processes (SM50) to ensure utilization stays below 80%.

    import pyrfc
import pandas as pd
import boto3
from concurrent.futures import ThreadPoolExecutor
from io import BytesIO
import pyarrow.parquet as pq
import pyarrow as pa

# Define partitions: (company code, fiscal year)
partitions = [
    ('1000', '2022'),
    ('1000', '2023'),
    ('1000', '2024'),
    ('2000', '2024'),
]

def extract_partition(bukrs, gjahr):
    """Extract BKPF for one (BUKRS, GJAHR) partition."""
    conn = pyrfc.Connection(
        ashost='sap-prod.company.com',
        sysnr='00', client='100',
        user='EXTRACT_BKPF', passwd='<password>'
    )
    
    result = conn.call(
        'RODPS_REPL_ODP_EXTRACT',
        SUBSCRIBER_NAME=f'PYTHON_BKPF_{bukrs}_{gjahr}',
        SUBSCRIBER_TYPE='2',
        SUBSCRIBER_PROCESS='FULL_LOAD',
        DATA_AREA='ABAP_CDS',
        ENTITY_NAME='I_AccountingDocument',
        MAX_RECORDS=500000,
        FIELDS=[{'FIELDNAME': '*'}],
        FILTERS=[
            {'NAME': 'BUKRS', 'SIGN': 'I', 'OPTION': 'EQ', 'LOW': bukrs},
            {'NAME': 'GJAHR', 'SIGN': 'I', 'OPTION': 'EQ', 'LOW': gjahr}
        ]
    )
    
    records = result.get('DATA', [])
    df = pd.DataFrame(records)
    print(f"Partition {bukrs}/{gjahr}: {len(df)} rows")
    
    # Write to S3
    s3 = boto3.client('s3')
    table = pa.Table.from_pandas(df)
    buf = BytesIO()
    pq.write_table(table, buf, compression='snappy')
    buf.seek(0)
    s3.put_object(
        Bucket='my-sap-landing-bucket',
        Key=f'sap/bkpf/{bukrs}/{gjahr}/bkpf.parquet',
        Body=buf.getvalue()
    )
    conn.close()
    return len(df)

# Run partitions in parallel (max 4 at a time)
with ThreadPoolExecutor(max_workers=4) as executor:
    futures = [executor.submit(extract_partition, bukrs, gjahr) 
              for bukrs, gjahr in partitions]
    total_rows = sum(f.result() for f in futures)

print(f"Total extracted: {total_rows} rows across {len(partitions)} partitions")

    Verify: Each partition extracts successfully. Total row count matches SE16N counts summed across all (BUKRS, GJAHR) combinations.

Troubleshooting
ADF pipeline runs but no data arrives in sink

Check that the Integration Runtime can reach both the SAP host and the sink. Confirm the source ODP dataset has a valid subscription key — open ODQMON on the SAP side and verify a subscription for your OData consumer appears. Also confirm the pipeline activity timeout is at least 4 hours for heavyweight tables.

Databricks JDBC read hangs or times out

RFC_READ_TABLE via JDBC is limited to ~500 MB result sets per call by default. For large tables, enable partition mode in the JDBC options (partitionColumn, lowerBound, upperBound, numPartitions) to split the read across parallel ABAP work processes. Check SM50 on the SAP side for stuck RFC work processes.

Fivetran sync paused with "schema drift" error

Fivetran detects column changes automatically but pauses the sync to avoid data loss. Go to the connector settings, review the new/removed columns, and resume. If the column was added to a custom Append Structure, confirm it appears in SE11 before resuming — Fivetran re-reads the schema from ABAP.

Airbyte stream returns 0 records on first sync

The ODP provider may require an init-delta run before returning data. In the Airbyte connection settings, trigger a full refresh for the first run. After the init completes, subsequent incremental syncs will return changes correctly. Confirm the ODP subscription appears in ODQMON after the first run.

Custom RFC call returns truncated rows

RFC_READ_TABLE row limit is 512 bytes per row by default. Columns with long text fields overflow. Switch to /SAPDS/RFC_READ_TABLE2 or a custom BAPI that returns the full row width. Alternatively, select fewer columns per call and join results in your pipeline.

Azure Data Lake Storage Gen2

ADLS is your landing zone — your tool writes Parquet, Delta, or Iceberg files here. Downstream engines (Synapse, Databricks, Power BI) read directly from these files.

Native SAP connector option

If your organisation has an ADF SAP connector license (SAP connector for Azure Data Factory), ADF writes directly to ADLS from SAP — no intermediate tool pipeline code needed. Confirm connector licensing with your Azure team before setting up the manual pipeline below.

Before your first extraction

  • Create an ADLS Gen2 storage account and container (e.g., sap-landing).
  • Register an Entra ID (Azure AD) app. Grant it Storage Blob Data Contributor on the container. Store tenant ID, client ID, and secret in your pipeline's secrets manager.
  • Agree on a partition path convention: sap/bkpf/rbukrs=1000/year=2024/. Partition columns must match the filter dimensions used in the SAP extraction.
  1. 1. Confirm the file landed after extraction

    After your tool runs, confirm the Parquet (or Delta/Iceberg) file lands in the container at the correct partition path. Use Azure Storage Explorer or the Azure portal. Prefer Parquet with Snappy compression; use Delta or Iceberg if your tool supports table formats.

    Verify: File visible at sap/bkpf/rbukrs=1000/year=2024/ in the container.

  2. 2. Validate row count from Synapse or Databricks

    Run a count query directly against the landed files. Compare to the SE16N count from the SAP-side steps — this is your reconciliation checkpoint before handing data to analytics.

    -- Synapse serverless / Databricks SQL
        SELECT COUNT(*) AS row_count
        FROM parquet.`abfss://sap-landing@account.dfs.core.windows.net/sap/bkpf/`;

    Verify: row_count matches SE16N.

Amazon S3

S3 is your landing zone — your tool writes Parquet files here. Downstream engines (Athena, Redshift Spectrum, Databricks on AWS) read directly from S3, or you COPY INTO Snowflake/Redshift from here.

Native SAP connector option

If your organisation has AWS AppFlow with SAP OData connector or a licensed SAP Integration Suite adapter, AppFlow can land data directly in S3 without tool pipeline code. Check your AWS and SAP BTP licensing before building the pipeline below.

Before your first extraction

  • Create an S3 bucket in your preferred region. Enable server-side encryption (SSE-S3 minimum) and block all public access.
  • Create an IAM role with s3:PutObject and s3:GetObject on the specific bucket/prefix ARN. Use instance profiles or IRSA — avoid long-lived access keys.
  • Agree on a landing prefix convention: sap/bkpf/rbukrs=1000/year=2024/.
  1. 1. Confirm files land and register in Glue

    After extraction, confirm files appear at the S3 prefix. Register the prefix as a Glue table pointing to Parquet files so Athena and Spark can discover it. If the next layer is Snowflake, create an external stage on this bucket path.

    aws s3 ls s3://your-bucket/sap/bkpf/ --recursive
        aws glue start-crawler --name sap-bkpf-crawler

    Verify: S3 listing returns non-zero bytes. Glue crawler status shows READY and created the table.

  2. 2. Validate row count in Athena

    Run a count in Athena and compare to SE16N. Athena scans Parquet efficiently when partition columns match the Hive-style path used for landing.

    SELECT COUNT(*) AS row_count
        FROM sap_landing.bkpf;

    Verify: Athena count matches SE16N. Downstream team can query the Glue table without moving files.

Snowflake

Snowflake reads Parquet files your tool landed in ADLS or S3. Two loading patterns: COPY INTO (batch, scheduled) or Snowpipe (continuous, file-triggered — runs automatically when a file lands, no scheduling needed).

Native SAP connector option — skip the steps below if licensed

If your organisation has a Snowflake Native SAP BDC Connector license, Snowflake connects directly to SAP — no tool pipeline code, no external stage, and no Parquet files needed. The connector extracts and loads natively. If this applies to you, configure the connector in Snowflake Partner Connect with your SAP RFC credentials and skip the COPY INTO / Snowpipe steps below.

Before your first extraction (file-based approach)

  • Create a dedicated database (SAP_LANDING), schema (RAW), and warehouse (SAP_LOAD_WH — X-Small for bulk loads).
  • Create a Snowflake storage integration (not access keys) for your ADLS container or S3 bucket. Create an external stage pointing to the landing path where your tool drops Parquet files.
  • Create the target table using the generated DDL below (derived from the table's column schema). Alternatively, let Snowflake auto-infer the schema via MATCH_BY_COLUMN_NAME on the first COPY INTO.
CREATE TABLE — Snowflake DDL 
-- Snowflake · BKPF · generated from column schema
CREATE OR REPLACE TABLE SAP_FI.BKPF (
  MANDT     VARCHAR(3) NOT NULL,
  BUKRS     VARCHAR(4) NOT NULL,
  BELNR     VARCHAR(10) NOT NULL,
  GJAHR     NUMBER(4) NOT NULL,
  BLART     VARCHAR(2),
  BLDAT     DATE,
  BUDAT     DATE,
  MONAT     NUMBER(2),
  CPUDT     DATE,
  WAERS     VARCHAR(5),
  KURSF     NUMBER(9, 2),
  XBLNR     VARCHAR(16),
  BKTXT     VARCHAR(25),
  STBLG     VARCHAR(10),
  AWTYP     VARCHAR(5),
  USNAM     VARCHAR(12),
  CONSTRAINT PK_BKPF PRIMARY KEY (MANDT, BUKRS, BELNR, GJAHR)
)
CLUSTER BY (MANDT, BUKRS);
  1. 1. Auto-create the target table on first load (optional)

    If you skipped the explicit CREATE TABLE above, Snowflake can infer the schema from the first Parquet file. Run once with VALIDATION_MODE for a dry-run:

    COPY INTO SAP_LANDING.RAW.bkpf
        FROM @your_stage/sap/bkpf/
        PATTERN='.*parquet'
        FILE_FORMAT=(TYPE=PARQUET)
        MATCH_BY_COLUMN_NAME=CASE_INSENSITIVE
        ON_ERROR=SKIP_FILE
        VALIDATION_MODE=RETURN_ERRORS;

    Tip: VALIDATION_MODE=RETURN_ERRORS inspects the Parquet schema without committing rows — confirm column mapping before the real load.

  2. 2. Load from external stage — COPY INTO or Snowpipe

    Choose one of two loading patterns:

    Batch (COPY INTO) — scheduled

    COPY INTO SAP_LANDING.RAW.bkpf
        FROM @your_stage/sap/bkpf/
        PATTERN='.*parquet'
        FILE_FORMAT=(TYPE=PARQUET);

    Run this on a Snowflake task or from your orchestrator after each extraction.

    Continuous (Snowpipe) — auto-ingest on file arrival

    CREATE PIPE sap_bkpf_pipe AS
        COPY INTO SAP_LANDING.RAW.bkpf
        FROM @your_stage/sap/bkpf/
        FILE_FORMAT=(TYPE=PARQUET);

    Enable auto-ingest on the stage — Snowpipe fires automatically when a file lands. Ideal for near-real-time SLT feeds.

    Verify: LIST @your_stage/sap/bkpf/ shows landed files. COPY INTO or Snowpipe status shows zero skipped files.

  3. 3. Validate row count after load

    Run a count against the loaded table and compare it to the SE16N count from the SAP-side steps:

    SELECT COUNT(*) AS row_count
        FROM SAP_LANDING.RAW.bkpf;
        
        -- If counts diverge, inspect load history
        SELECT *
        FROM INFORMATION_SCHEMA.LOAD_HISTORY
        WHERE TABLE_NAME = 'bkpf'
          AND ERROR_COUNT > 0
        ORDER BY LAST_LOAD_TIME DESC;

    Verify: row_count matches SE16N. Zero error rows in load history.

BigQuery

    Verify: Count matches SE16N. Partition metadata visible in BigQuery console.

Microsoft Fabric / OneLake

Other landing zone

Whatever your sink: it receives Parquet, Delta, or Iceberg files from your tool. Ensure your downstream system can read one of these formats efficiently before landing data in it.

Before your first extraction

  • Confirm your sink can read Parquet (most portable), Delta, or Iceberg. Avoid CSV for tables over 10M rows.
  • Agree on a partitioned path convention with the analytics team: sap/bkpf/rbukrs=1000/year=2024/run=TIMESTAMP/.
  • Test a small write to your sink before running a full extraction — confirm schema mapping, encoding, and permissions.
  1. 1. Confirm file lands at the correct path

    After extraction, verify the file appears at the agreed partition path. Check file size is reasonable for the expected row count (Parquet with Snappy is typically 100–300 bytes per row for journal data).

    Verify: File is present and non-zero. Schema is readable by your downstream tool.

  2. 2. Row-count reconciliation is mandatory

    Always compare the row count from your sink against the SE16N count from the SAP-side steps. A mismatch means the extraction dropped rows or the filter scopes differ — investigate before handing data to analytics. Automate this check in your pipeline so it fails the run on divergence.

    Verify: Your pipeline has an automated row-count check that fails the run if counts diverge beyond a tolerance threshold.

Troubleshooting
Files land in ADLS but Synapse/Databricks can't read them

Confirm the storage account has Hierarchical Namespace (HNS) enabled — standard ADLS Blob storage doesn't support ABFSS URIs. Check that the service principal has Storage Blob Data Reader on the container, not just Contributor on the storage account. Verify the Parquet file isn't corrupted by running a quick count from Azure Storage Explorer's preview.

S3 upload succeeds but Glue crawler shows 0 records

The Parquet file may be in a nested folder that doesn't match the Glue partition pattern. Confirm the S3 prefix uses Hive-style partitions (e.g., rbukrs=1000/year=2024/). Run MSCK REPAIR TABLE your_table in Athena after re-crawling to register new partitions. Check that the Glue IAM role has s3:GetObject on the exact bucket ARN, not just the bucket root.

COPY INTO shows 0 rows loaded or "file format mismatch"

Confirm the external stage points to the correct S3/ADLS prefix — run LIST @your_stage/sap/slug/ and verify files appear. If using MATCH_BY_COLUMN_NAME, the Parquet column names must exactly match the table column names (case-sensitive). For Snowpipe, check pipe status with SELECT SYSTEM$PIPE_STATUS('your_pipe') and inspect the error log table.

BigQuery external table returns stale or missing data

External tables in BigQuery always reflect the current state of the underlying GCS files — if new files were added after the external table was created, they are automatically included (no refresh needed). If data appears stale, confirm the GCS bucket path in the external table definition matches where your tool is writing. For persistent tables loaded via a load job, re-run the load after each extraction.

Fabric Lakehouse shortcut shows empty or stale data

OneLake shortcuts reflect ADLS in near-real time but may cache metadata. Click "Refresh" on the Lakehouse table in the Fabric UI. If the Delta table shows stale data, run DESCRIBE HISTORY your_table in a Fabric notebook to confirm the latest Delta commit was written. For Parquet shortcuts, the file must be at exactly the path the shortcut points to — a new subfolder will not be automatically included.

Databricks row count doesn't match SE16N after writing

Check for concurrent writes — if another pipeline wrote to the same Delta table path during the extraction, rows may be duplicated or the table may be in an inconsistent state. Run DESCRIBE HISTORY sap_landing.raw.slug to see all commits. Use VACUUM followed by RESTORE if you need to roll back. For initial loads, use OVERWRITE mode to ensure idempotency.

BigQuery external table on GCS returns permission denied

The BigQuery service account needs Storage Object Viewer on the GCS bucket — not on the project. Add the IAM binding at the bucket level, not the project level. If using a BigLake Iceberg table, the service account also needs BigLake connection permissions. Run bq show --format=prettyjson your_dataset.your_table to confirm the external table's connection is active.

Row count in target doesn't match SE16N

A mismatch typically means the filter scope differs between SAP and your sink query. Confirm the same RBUKRS, RYEAR (and POPER if applicable) filters are used in both SE16N and the count query against your sink. Also check for type conversion issues — NUMC fields stored as integers in the target may not match the string representation in SE16N's filter.
