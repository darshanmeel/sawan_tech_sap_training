Extract VBAK with Delta and Z-Field (Intermediate)

SAP-side intermediate guide for VBAK extraction with delta processing and a custom Z-field. Covers ODP delta queue mechanics in ODQMON, CDS extension view creation in SE80 for Z-field inclusion, and checking delta subscription health.

Estimated time
60 minutes
Destinations
Generic
Extractors
ODP

Four things to confirm before any tool runs

The checks below, in order, are the difference between a smooth extraction and a multi-day incident. Each one is one sentence; expand for the how.

  1. 01 · Licence

    Confirm your extraction method is covered by your installed SAP licence before anyone designs anything.

    How to check

    Runtime licences cover ODP through OData. SLT replication requires a Full Use licence. Ask your SAP licensing contact; do not assume. See the dedicated article on Runtime vs Full Use, and note that SAP Note 3255746 prohibits third-party tools from using ODP via RFC — compliant paths are ODP over OData or SLT. [NEEDS_SAP_CITATION]

  2. 02 · Released CDS view

    Every table covered here has an extraction-safe released CDS view; confirm it exists before building.

    How to check

    Open the view in SE80 (or in ADT) and confirm the header annotation @Analytics.dataExtract: true is present. If it is missing, the view is not extractable — raw-table reads are not the fix.

  3. 03 · Authorisations

    A System-type technical user with the minimum authorisation objects for your chosen method — no named users, no wildcards.

    How to check

    Create a System user via SU01 and assign a role via PFCG. For ODP: S_RFC (RFC_TYPE=FUNC) and S_ODP_READ (ODPSOURCE=ABAP_CDS). For SLT your Basis team also needs authorisation in LTCO and LTRS. For RFC direct add S_TABU_DIS scoped to the table's authorisation group — never wildcard it.

  4. 04 · Row-count baseline

    Use SE16N to measure the intended partition before extraction — you cannot reconcile without this number.

    How to check

    Open SE16N, apply the same filter your extraction will use, and record the count. Under 100M rows is usually safe for a single ODP partition; larger volumes need further splitting or SLT. A count in the billions means the filter is misconfigured — rethink the partition before running anything.

License check

All license types. CDS extension views are application-layer. License trap: Runtime vs Full Use →

ODP (Operational Data Provisioning) is permitted under both Runtime and Full Use SAP licenses when you access data through published CDS views via the OData API.

SAP Note 3255746 (Feb 2024): Third-party tools that call ODP via RFC (ADF SAP CDC, Qlik, Fivetran ODP, Informatica ODP, etc.) are no longer permitted. Use ODP via OData, or switch to SLT. Verify with your licensing contact. Full note (SAP login required). [NEEDS_SAP_CITATION — confirm current note state]

SLT (SAP Landscape Transformation) requires a Full Use SAP license. It is not permitted under Runtime licenses. Verify in SLICENSE that "SLT Landscape Transformation" is listed as Active, Type = Full Use before you configure anything in LTCO.

License trap: Runtime vs Full Use →

Direct RFC extraction (e.g., RFC_READ_TABLE) is generally permitted for read-only access under Runtime and Full Use licenses, but is not recommended for production use — it bypasses authorization checks that released CDS views enforce and is subject to SAP Note 3255746 restrictions when wrapped by third-party ODP connectors. Overview

Scenario

Your analytics team now needs near-real-time sales data. Daily delta extracts are sufficient — you do not need sub-minute updates, but every morning's refresh should catch all yesterday's new and changed orders. Your company also added a custom Z-field ZZ_REGION to sales documents for regional analysis.

This walkthrough covers the SAP-side changes: confirming delta support, creating a CDS extension view for ZZ_REGION, and verifying that the delta queue in ODQMON is healthy. Your tool team configures the delta poll schedule.

What you have confirmed

The SAP side supports delta extraction for VBAK, the Z-field is exposed through a CDS extension view, and the ODQMON subscription is healthy. Your tool team can now configure daily delta polls. When you need sub-minute real-time updates — for example, to feed a live sales dashboard — move to the Expert walkthrough and SLT.

Before you start

  • Completed VBAK Beginner walkthrough
  • Understanding of ODP delta concepts from the ODP glossary entry
0 of 5 steps complete
  1. Step 1

    Verify I_SalesDocument supports delta extraction

    Open I_SalesDocument in SE80 and inspect the delta annotation. For ODP delta to work, the view must carry @Analytics.dataExtraction.delta.changeDataCapture.automatic: true.

    Do this in SAP
    Open transaction SE80 (opens in new tab) and navigate to Repository Browser → I_SalesDocument → Annotations.
    Verify
    I_SalesDocument has @Analytics.dataExtract: true. Delta annotation is present or Basis team confirms delta is supported.
  2. Step 2

    Check for the custom Z-field ZZ_REGION in VBAK

    Your S/4HANA system has a custom Z-field ZZ_REGION on VBAK (added via an Append Structure). Confirm this field exists in the table definition using SE11 before building a CDS extension view to expose it.

    Do this in SAP
    Open transaction SE11 (opens in new tab) and navigate to Data Dictionary → Database Tables → VBAK → Display → Fields.
    Verify
    ZZ_REGION appears in the VBAK field list under an Append Structure entry.
    Why this matters

    Z-fields added via Append Structures are part of the physical table but are not included in standard released CDS views. You must expose them explicitly through a CDS extension view to make them available for ODP extraction.

  3. Step 3

    Create a CDS extension view to expose ZZ_REGION

    In SE80 or ABAP Development Tools (ADT), create a CDS extension view that adds ZZ_REGION to I_SalesDocument. The extension view must not modify the released view — it extends it.

    Do this in SAP
    Open transaction SE80 (opens in new tab) and navigate to Repository Browser → Create → CDS View Extension.
    Verify
    Extension view ZI_SalesDocument_Ext is active in SE80. Activating it does not raise errors or modify I_SalesDocument itself.
  4. Step 4

    Monitor the ODP delta subscription in ODQMON

    After your extraction tool registers and runs its init-delta, check ODQMON. Confirm the subscription exists and is in an active (not stuck) state. Note: the first delta after init returns 0 records — this is normal per SAP Note 2884410.

    Do this in SAP
    Open transaction ODQMON (opens in new tab) and navigate to Subscriptions → Context: ABAP_CDS → I_SalesDocument (or ZI_SalesDocument_Ext).
    Verify
    Subscription is active. After init run, delta queue shows 0 pending records (expected). After subsequent changes in SAP, queue shows pending records matching number of changed documents.
  5. Step 5

    Validate Z-field data in SE16N

    Before handing data to your analytics team, verify that ZZ_REGION values look correct. Use SE16N to browse VBAK and check ZZ_REGION values for a sample of document numbers. Then confirm those same values appear in your target system for the same VBELN keys.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → VBAK → Show ZZ_REGION for sample VBELN.
    Verify
    ZZ_REGION values in your target match ZZ_REGION values in SE16N for the same VBELN records.

Troubleshooting

ZZ_REGION is NULL in extraction output

The CDS extension view may not be active or may not be the source your tool is pointing at. Confirm the extension view is active in SE80 and that your tool's ODP source is configured to use ZI_SalesDocument_Ext (not the base I_SalesDocument). Also confirm the ABAP transport containing the extension view is in the production system.

Delta subscriptions accumulate in ODQMON without being consumed

Check whether your extraction scheduler is running. Stale subscriptions that accumulate more than a few days of delta entries can cause performance issues in ODQMON. If the subscriber is no longer active, delete it in ODQMON to prevent queue buildup.
