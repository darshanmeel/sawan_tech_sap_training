Extract MARA with Z-Fields via CDS Extension (Intermediate)

SAP-side intermediate guide for MARA extraction with custom Z-fields. Covers identifying Z-fields in SE11 Append Structures, creating a CDS extension view in SE80, confirming the extension view activates cleanly, and verifying Z-field values with SE16N.

Estimated time
60 minutes
Destinations
Generic
Extractors
ODP

SAP-side preflight

Before any tool touches your SAP system, you need four things confirmed on the SAP side. Skipping these steps is the most common cause of extraction failures, performance incidents, and licensing surprises.

  1. License — confirm your extraction method is covered.

    Runtime vs Full Use is the first decision. ODP-based extraction (via ODP) is permitted under Runtime licenses. SLT replication requires a Full Use license. Verify with your SAP licensing contact before you design anything — not the day before go-live.

    SAP Note 3255746 (Feb 2024): Use of ODP RFC function modules by third-party tools is no longer permitted. If your tool uses ODP via RFC (ADF SAP CDC, Qlik, Fivetran ODP, etc.), verify your contract terms. Compliant alternatives are ODP via OData or SLT replication. Full note at support.sap.com/notes/3255746 (SAP login required). [NEEDS_SAP_CITATION — verify current note version]

  2. Released CDS view — confirm the extraction-safe view exists.

    Every table covered in this Academy has a SAP-released CDS view annotated for extraction. Open SE80 and search for the view name before you build anything. Confirm the annotation @Analytics.dataExtract: true is present. If the view is absent or the annotation is missing, stop — raw table extraction is not the solution.

  3. Authorizations — grant the extraction user minimum permissions.

    Your extraction user needs three authorization objects. Use SU01 to create a System-type (non-dialog) user, then use PFCG to assign a role containing:

    • S_RFC — allows RFC calls (set RFC_TYPE=FUNC)
    • S_ODP_READ — allows reading from ODP contexts (set ODPSOURCE=ABAP_CDS for CDS-based extraction)
    • S_TABU_DIS — table display access for the relevant authorization group

    For SLT-based extraction, your Basis team will also need authorization in LTCO and LTRS. [NEEDS_SAP_CITATION — confirm exact auth objects for LTCO/LTRS]

  4. Row-count context — understand data volumes before you run.

    Use SE16N to count rows in the target table with your intended filter (company code, fiscal year, material type, etc.) before you start the extraction. This gives you:

    • A baseline to reconcile against after the extract.
    • A sense of whether your partition is safe to extract (< 100M rows is generally safe for ODP; larger partitions may need further splitting or SLT).
    • Early warning if a filter is misconfigured (count returns billions — rethink the partition).

These four checks take 15–30 minutes on a first walkthrough, and under 5 minutes once you know your system. They are the difference between a smooth extraction and a multi-day incident.

License check

All licenses. CDS extension views are application-layer. License trap: Runtime vs Full Use →

ODP (Operational Data Provisioning) is permitted under both Runtime and Full Use SAP licenses when you access data through published CDS views via the OData API.

SAP Note 3255746 (Feb 2024): Third-party tools that call ODP via RFC (ADF SAP CDC, Qlik, Fivetran ODP, Informatica ODP, etc.) are no longer permitted. Use ODP via OData, or switch to SLT. Verify with your licensing contact. Full note (SAP login required). [NEEDS_SAP_CITATION — confirm current note state]

SLT (SAP Landscape Transformation) requires a Full Use SAP license. It is not permitted under Runtime licenses. Verify in SLICENSE that "SLT Landscape Transformation" is listed as Active, Type = Full Use before you configure anything in LTCO.

License trap: Runtime vs Full Use →

Direct RFC extraction (e.g., RFC_READ_TABLE) is generally permitted for read-only access under Runtime and Full Use licenses, but is not recommended for production use — it bypasses authorization checks that released CDS views enforce and is subject to SAP Note 3255746 restrictions when wrapped by third-party ODP connectors. Overview

Scenario

Your company extended MARA with Z-fields for brand and sustainability tracking. You need these fields in the data warehouse without modifying the released CDS view.

This walkthrough covers the SAP-side work: identifying Z-fields in SE11, creating a CDS extension view, and verifying the Z-field data is correctly exposed. Your tool team points the extraction at the extension view instead of the base I_Product view.

What you have confirmed

You have created and activated a CDS extension view that exposes MARA Z-fields, confirmed the Z-field values look correct in SE80 data preview, and cross-checked against SE16N. The data warehouse now includes brand and sustainability dimensions in the next extraction run.

Before you start

  • Completed MARA Beginner walkthrough
  • SE80 familiarity — you can open and read a CDS view
  • Access to SE11 to inspect table field lists
0 of 5 steps complete
  1. Step 1

    Identify Z-fields in MARA via SE11

    Your MARA table carries custom Z-fields added through Append Structures. Open SE11, display the MARA table definition, and scroll to the field list. Append Structure fields are grouped separately. In this walkthrough, the target Z-fields are ZZ_BRAND (20-char) and ZZ_SUSTAINABILITY (1-char).

    Do this in SAP
    Open transaction SE11 (opens in new tab) and navigate to Data Dictionary → Database Tables → MARA → Display → Fields → Find Z*.
    Verify
    ZZ_BRAND (CHAR 20) and ZZ_SUSTAINABILITY (CHAR 1) appear in the MARA field list under an Append Structure entry.
    Why this matters

    Z-fields in Append Structures are part of the physical table but are not visible in the standard released CDS view. You must expose them explicitly. Knowing the exact field names and data types before creating the extension view prevents activation errors.

  2. Step 2

    Check that the Append Structure has data

    Use SE16N to browse MARA rows and confirm that ZZ_BRAND contains expected values. If all values are empty (initial), the Z-field may not yet be populated from the upstream business process. Extracting empty Z-fields is harmless but produces no analytical value.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → MARA → Add column ZZ_BRAND → Browse sample rows.
    Verify
    ZZ_BRAND contains non-initial values for at least a subset of materials. You know what expected values look like (e.g., 'PREMIUM', 'STANDARD').
  3. Step 3

    Create a CDS extension view to expose Z-fields

    In SE80 or ABAP Development Tools (ADT), create an extension view that adds ZZ_BRAND and ZZ_SUSTAINABILITY to I_Product. The extension view must carry @AccessControl.authorizationCheck: #CHECK — without it, unauthorized users can read the Z-fields. Activate the view and verify it has no syntax errors.

    Do this in SAP
    Open transaction SE80 (opens in new tab) and navigate to Repository Browser → Create → CDS View Extension.
    Verify
    ZPROD_ZFIELDS is active in SE80. Activation log shows no errors. The base view I_Product is not modified.
  4. Step 4

    Verify the extension view returns Z-field data

    From SE80, run a data preview on ZPROD_ZFIELDS and confirm ZZ_BRAND and ZZ_SUSTAINABILITY values appear alongside standard MARA fields.

    Do this in SAP
    Open transaction SE80 (opens in new tab) and navigate to ZPROD_ZFIELDS → Data Preview.
    Verify
    Preview shows MATNR, MTART, ZZ_BRAND, ZZ_SUSTAINABILITY columns with expected values.
  5. Step 5

    Cross-check Z-field values with SE16N after extraction

    After extraction, pick a sample of MATNR values and compare ZZ_BRAND values in your target system against the SE16N MARA values for the same material numbers.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → MARA → Show ZZ_BRAND for sample MATNR.
    Verify
    ZZ_BRAND values in target match ZZ_BRAND values in SE16N for the same MATNR records.

Troubleshooting

CDS extension view activation fails with syntax error

Check the exact syntax against your ABAP CDS documentation. Common issues: missing @AccessControl annotation, incorrect field name (must match SE11 exactly), using MARA. prefix when the table alias is different in I_Product. Check the activation log in SE80 for the specific error message.

Z-field values are empty in extraction output but not in SE16N

The extension view may be pointing to the wrong table alias. In SE80, inspect I_Product's FROM clause to find the alias used for MARA rows. Use that alias in the extension view (e.g., if I_Product uses alias 'prod', use 'prod.ZZ_BRAND' not 'mara.ZZ_BRAND').
