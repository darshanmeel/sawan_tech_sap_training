Extract LFA1 with Hourly Delta (Intermediate)

SAP-side intermediate guide for LFA1 extraction with ODP delta. Covers creating an ODP subscription in ODQMON, understanding subscription lifecycle, validating delta behavior with manual SAP edits, and monitoring queue health.

Estimated time
55 minutes
Destinations
Generic
Extractors
ODP

SAP-side preflight

Before any tool touches your SAP system, you need four things confirmed on the SAP side. Skipping these steps is the most common cause of extraction failures, performance incidents, and licensing surprises.

  1. License — confirm your extraction method is covered.

    Runtime vs Full Use is the first decision. ODP-based extraction (via ODP) is permitted under Runtime licenses. SLT replication requires a Full Use license. Verify with your SAP licensing contact before you design anything — not the day before go-live.

    SAP Note 3255746 (Feb 2024): Use of ODP RFC function modules by third-party tools is no longer permitted. If your tool uses ODP via RFC (ADF SAP CDC, Qlik, Fivetran ODP, etc.), verify your contract terms. Compliant alternatives are ODP via OData or SLT replication. Full note at support.sap.com/notes/3255746 (SAP login required). [NEEDS_SAP_CITATION — verify current note version]

  2. Released CDS view — confirm the extraction-safe view exists.

    Every table covered in this Academy has a SAP-released CDS view annotated for extraction. Open SE80 and search for the view name before you build anything. Confirm the annotation @Analytics.dataExtract: true is present. If the view is absent or the annotation is missing, stop — raw table extraction is not the solution.

  3. Authorizations — grant the extraction user minimum permissions.

    Your extraction user needs three authorization objects. Use SU01 to create a System-type (non-dialog) user, then use PFCG to assign a role containing:

    • S_RFC — allows RFC calls (set RFC_TYPE=FUNC)
    • S_ODP_READ — allows reading from ODP contexts (set ODPSOURCE=ABAP_CDS for CDS-based extraction)
    • S_TABU_DIS — table display access for the relevant authorization group

    For SLT-based extraction, your Basis team will also need authorization in LTCO and LTRS. [NEEDS_SAP_CITATION — confirm exact auth objects for LTCO/LTRS]

  4. Row-count context — understand data volumes before you run.

    Use SE16N to count rows in the target table with your intended filter (company code, fiscal year, material type, etc.) before you start the extraction. This gives you:

    • A baseline to reconcile against after the extract.
    • A sense of whether your partition is safe to extract (< 100M rows is generally safe for ODP; larger partitions may need further splitting or SLT).
    • Early warning if a filter is misconfigured (count returns billions — rethink the partition).

These four checks take 15–30 minutes on a first walkthrough, and under 5 minutes once you know your system. They are the difference between a smooth extraction and a multi-day incident.

License check

All licenses. ODP is application-layer. License trap: Runtime vs Full Use →

ODP (Operational Data Provisioning) is permitted under both Runtime and Full Use SAP licenses when you access data through published CDS views via the OData API.

SAP Note 3255746 (Feb 2024): Third-party tools that call ODP via RFC (ADF SAP CDC, Qlik, Fivetran ODP, Informatica ODP, etc.) are no longer permitted. Use ODP via OData, or switch to SLT. Verify with your licensing contact. Full note (SAP login required). [NEEDS_SAP_CITATION — confirm current note state]

SLT (SAP Landscape Transformation) requires a Full Use SAP license. It is not permitted under Runtime licenses. Verify in SLICENSE that "SLT Landscape Transformation" is listed as Active, Type = Full Use before you configure anything in LTCO.

License trap: Runtime vs Full Use →

Direct RFC extraction (e.g., RFC_READ_TABLE) is generally permitted for read-only access under Runtime and Full Use licenses, but is not recommended for production use — it bypasses authorization checks that released CDS views enforce and is subject to SAP Note 3255746 restrictions when wrapped by third-party ODP connectors. Overview

Scenario

Your procurement dashboard needs hourly vendor updates. You will implement ODP delta extraction for LFA1, test it with a controlled vendor change, and set up weekly full-load reconciliation to catch any missed changes.

What you have confirmed

The SAP delta mechanism for I_Supplier is working. You can see delta changes appear in ODQMON within minutes of a vendor change, and you understand the subscription lifecycle and queue retention limits. Vendor master is now near-real-time.

Before you start

  • Completed LFA1 Beginner walkthrough
  • Understanding of ODP delta concepts from the ODP glossary entry
0 of 5 steps complete
  1. Step 1

    Verify I_Supplier has delta annotation in SE80

    Open I_Supplier in SE80 and confirm it carries the delta changeDataCapture annotation required for incremental extraction. If the annotation is absent on your release, you will need to run full-load extractions on every cycle — which is still viable for vendor master given its small size, but is less efficient.

    Do this in SAP
    Open transaction SE80 (opens in new tab) and navigate to Repository Browser → I_Supplier → Annotations.
    Verify
    I_Supplier has @Analytics.dataExtract: true. Delta annotation present or confirmed by Basis team.
  2. Step 2

    Create an ODP subscription in ODQMON

    Your extraction tool creates subscriptions automatically when it first connects. But you can also inspect existing subscriptions manually via ODQMON to understand their state. After the tool runs its first extraction, navigate to the Subscriptions tab, filter by ABAP_CDS context, and find the subscription name your tool created.

    Do this in SAP
    Open transaction ODQMON (opens in new tab) and navigate to Subscriptions → Context: ABAP_CDS → I_Supplier.
    Verify
    Subscription for I_Supplier appears after the tool's first run. Status is active.
  3. Step 3

    Test delta behavior with a manual vendor change

    To confirm the delta mechanism is working, make a controlled change in SAP: update a vendor record in SU01 or in the vendor master transaction (FK02). After the change, wait 5 minutes and check ODQMON. The subscription should show 1 pending record in the delta queue. Then run the delta extraction and confirm that 1 record is fetched and the queue returns to 0.

    Do this in SAP
    Open transaction FK02 (opens in new tab) and navigate to Vendor → Change → General Data.
    Verify
    After manual vendor change, ODQMON delta queue for I_Supplier shows 1 pending record. After delta extraction, queue returns to 0.
    Why this matters

    Testing delta with a controlled change before relying on it for production data is essential. It confirms the delta mechanism works end-to-end on your specific system and release.

  4. Step 4

    Understand ODP subscription lifecycle and queue limits

    ODP subscriptions have a finite queue depth. If the subscriber does not consume the queue within a defined window (typically 48 hours by default, configurable by Basis), the queue entries expire. A subscriber that falls behind more than this window will miss changes and may need a full-load re-init. Ask your Basis team what the queue retention is configured to on your system. [NEEDS_SAP_CITATION — confirm default queue retention settings]

    Do this in SAP
    Open transaction ODQMON (opens in new tab) and navigate to Subscriptions → View Queue Depth and Age.
    Verify
    ODQMON shows queue depth for the LFA1 subscription. You understand the retention window configured on your system.
  5. Step 5

    Set up weekly full-load reconciliation

    Even with hourly delta, run a weekly full-load count comparison. Use SE16N to count LFA1 rows in SAP. Compare to your target. If the counts diverge significantly, the delta mechanism may have missed changes. Trigger a full re-init if needed.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → LFA1 → Count.
    Verify
    Weekly SE16N count matches target count within a small margin for in-flight updates.

Troubleshooting

Delta misses some vendor updates

ODP queue may have expired entries if extraction was delayed more than the queue retention window. Re-run a full-load to catch missing data. Ask Basis team to increase queue retention if hourly polling is not feasible.

ODQMON shows subscription in ERROR state

The extraction process may have failed mid-run and left the subscription in a bad state. Reset the subscription in ODQMON (right-click → Reset). Re-run the extraction. If the error persists, review the extraction tool's connection logs and check SM21 (system log) for RFC errors.
