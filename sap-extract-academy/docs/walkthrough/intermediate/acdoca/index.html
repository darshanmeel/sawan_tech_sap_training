Extract ACDOCA One Company/Year via ODP (Intermediate)

SAP-side intermediate guide for ACDOCA extraction. Focuses on confirming the I_JournalEntryItem CDS view, understanding ODP delta queue behavior, monitoring with ODQMON, and SE16N reconciliation for a single fiscal year partition.

Estimated time
90 minutes
Destinations
Generic
Extractors
ODP

SAP-side preflight

Before any tool touches your SAP system, you need four things confirmed on the SAP side. Skipping these steps is the most common cause of extraction failures, performance incidents, and licensing surprises.

  1. License — confirm your extraction method is covered.

    Runtime vs Full Use is the first decision. ODP-based extraction (via ODP) is permitted under Runtime licenses. SLT replication requires a Full Use license. Verify with your SAP licensing contact before you design anything — not the day before go-live.

    SAP Note 3255746 (Feb 2024): Use of ODP RFC function modules by third-party tools is no longer permitted. If your tool uses ODP via RFC (ADF SAP CDC, Qlik, Fivetran ODP, etc.), verify your contract terms. Compliant alternatives are ODP via OData or SLT replication. Full note at support.sap.com/notes/3255746 (SAP login required). [NEEDS_SAP_CITATION — verify current note version]

  2. Released CDS view — confirm the extraction-safe view exists.

    Every table covered in this Academy has a SAP-released CDS view annotated for extraction. Open SE80 and search for the view name before you build anything. Confirm the annotation @Analytics.dataExtract: true is present. If the view is absent or the annotation is missing, stop — raw table extraction is not the solution.

  3. Authorizations — grant the extraction user minimum permissions.

    Your extraction user needs three authorization objects. Use SU01 to create a System-type (non-dialog) user, then use PFCG to assign a role containing:

    • S_RFC — allows RFC calls (set RFC_TYPE=FUNC)
    • S_ODP_READ — allows reading from ODP contexts (set ODPSOURCE=ABAP_CDS for CDS-based extraction)
    • S_TABU_DIS — table display access for the relevant authorization group

    For SLT-based extraction, your Basis team will also need authorization in LTCO and LTRS. [NEEDS_SAP_CITATION — confirm exact auth objects for LTCO/LTRS]

  4. Row-count context — understand data volumes before you run.

    Use SE16N to count rows in the target table with your intended filter (company code, fiscal year, material type, etc.) before you start the extraction. This gives you:

    • A baseline to reconcile against after the extract.
    • A sense of whether your partition is safe to extract (< 100M rows is generally safe for ODP; larger partitions may need further splitting or SLT).
    • Early warning if a filter is misconfigured (count returns billions — rethink the partition).

These four checks take 15–30 minutes on a first walkthrough, and under 5 minutes once you know your system. They are the difference between a smooth extraction and a multi-day incident.

License check

ODP extraction via OData is permitted under Runtime and Full Use licenses. License trap: Runtime vs Full Use →

ODP (Operational Data Provisioning) is permitted under both Runtime and Full Use SAP licenses when you access data through published CDS views via the OData API.

SAP Note 3255746 (Feb 2024): Third-party tools that call ODP via RFC (ADF SAP CDC, Qlik, Fivetran ODP, Informatica ODP, etc.) are no longer permitted. Use ODP via OData, or switch to SLT. Verify with your licensing contact. Full note (SAP login required). [NEEDS_SAP_CITATION — confirm current note state]

SLT (SAP Landscape Transformation) requires a Full Use SAP license. It is not permitted under Runtime licenses. Verify in SLICENSE that "SLT Landscape Transformation" is listed as Active, Type = Full Use before you configure anything in LTCO.

License trap: Runtime vs Full Use →

Direct RFC extraction (e.g., RFC_READ_TABLE) is generally permitted for read-only access under Runtime and Full Use licenses, but is not recommended for production use — it bypasses authorization checks that released CDS views enforce and is subject to SAP Note 3255746 restrictions when wrapped by third-party ODP connectors. Overview

Scenario

Your finance team needs ACDOCA data for year-end close analytics. You will extract one company code, one fiscal year — enough for meaningful analysis without hitting system limits. This walkthrough focuses on the SAP-side checks: confirming delta capability, monitoring ODP subscriptions, and understanding the first-delta behavior that surprises most teams on their first run.

Estimated time: 90 minutes.

What you have confirmed

You have verified that I_JournalEntryItem supports delta extraction on your system, understand that the first ODP delta returns 0 records (by design), and have a reconciliation baseline. When you need more years or company codes, the Expert walkthrough scales these patterns via SLT.

Before you start

  • Completed any beginner walkthrough
  • Understanding of ACDOCA table structure from the ACDOCA beginner walkthrough or table overview
  • SE16N shows you know your partition row count
0 of 5 steps complete
  1. Step 1

    Confirm ACDOCA partition size in SE16N

    Before running any extraction, use SE16N to count rows for your intended partition. Filter on RBUKRS (company code) and RYEAR (fiscal year). If the count exceeds 500M rows for a single partition, move to the Expert walkthrough — that volume requires SLT or sub-partition strategies not covered here.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → ACDOCA → Filter RBUKRS='1000' AND RYEAR=2024 → Count.
    Verify
    Row count is below 500M. Write it down for reconciliation.
  2. Step 2

    Confirm I_JournalEntryItem CDS view and delta annotation

    In SE80, open I_JournalEntryItem and inspect the annotations. For delta extraction you need @Analytics.dataExtraction.delta.changeDataCapture.automatic: true. If that annotation is absent, the view supports only full-load extraction — delta will not work.

    Do this in SAP
    Open transaction SE80 (opens in new tab) and navigate to Repository Browser → I_JournalEntryItem → Annotations.
    Verify
    I_JournalEntryItem has @Analytics.dataExtract: true. Check for the delta changeDataCapture annotation. If absent, confirm with your Basis team whether a support package is needed to enable it.
    Why this matters

    If you run a delta extraction against a view that lacks the CDC annotation, ODP returns all rows on every run — the same behavior as a full load but without you realising it. This leads to duplicate data in your target.

  3. Step 3

    Monitor the ODP subscription in ODQMON

    After your extraction tool registers, open ODQMON and navigate to the ABAP_CDS context. Find the subscription for I_JournalEntryItem. Note: after an init-delta run, the first delta request returns 0 records. This is correct ODP behavior — the delta mechanism captures changes from the init point forward, not from before it. SAP Note 2884410 documents this.

    Do this in SAP
    Open transaction ODQMON (opens in new tab) and navigate to Subscriptions → Context: ABAP_CDS → I_JournalEntryItem.
    Verify
    Subscription for I_JournalEntryItem appears. After init, delta queue shows 0 pending records — this is expected.
  4. Step 4

    Reconcile extracted row count against SE16N

    After the full-load extraction completes, return to SE16N and recount ACDOCA rows for the same RBUKRS and RYEAR partition. The count in your target system should match within a small window for in-flight postings.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → ACDOCA → Filter RBUKRS='1000' AND RYEAR=2024 → Count.
    Verify
    Target row count matches SE16N count within a tolerable margin for in-flight transactions.
  5. Step 5

    Add posting period filter if the partition is still too large

    If SE16N shows your RBUKRS+RYEAR partition is too large (e.g., 200M+ rows) for a safe ODP extraction, add a posting period filter. In SE16N, filter on POPER (posting period, values 001–012 for months) to count rows per quarter. Then extract one quarter at a time.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → ACDOCA → Filter RBUKRS='1000' AND RYEAR=2024 AND POPER='001' → Count.
    Verify
    Each quarterly partition (e.g., POPER='001' through '003') has a manageable row count. Adjust partition granularity accordingly.

Troubleshooting

ODP extraction times out after 2 hours

The partition is more than 500M rows. Use the Expert walkthrough which covers SLT parallel readers for ACDOCA at that scale.

First delta run returns 0 records — is this a bug?

No. This is documented ODP behavior per SAP Note 2884410. After an init-delta, the first delta request returns 0 records because no changes have occurred since the init point. Subsequent delta runs will return only changed records.

ODQMON shows subscription stuck in 'FETCHING' state

The extraction process is still running or has hung. Check SM50 for active ABAP processes owned by your extraction user. If the process is not active, the subscription may be corrupted. Reset it in ODQMON and restart the extraction.
