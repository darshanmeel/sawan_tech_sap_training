Extract BKPF + BSEG at Enterprise Scale (Expert)

SAP-side expert guide for extracting BKPF (accounting document headers) and BSEG (line items) at enterprise scale via SLT. Covers BKPF/BSEG cardinality, LTCO configuration for both tables, LTRS parallel reader setup, and ODQMON join-coherence monitoring.

Estimated time
120 minutes
Destinations
Generic
Extractors
SLT

SAP-side preflight

Before any tool touches your SAP system, you need four things confirmed on the SAP side. Skipping these steps is the most common cause of extraction failures, performance incidents, and licensing surprises.

  1. License — confirm your extraction method is covered.

    Runtime vs Full Use is the first decision. ODP-based extraction (via ODP) is permitted under Runtime licenses. SLT replication requires a Full Use license. Verify with your SAP licensing contact before you design anything — not the day before go-live.

    SAP Note 3255746 (Feb 2024): Use of ODP RFC function modules by third-party tools is no longer permitted. If your tool uses ODP via RFC (ADF SAP CDC, Qlik, Fivetran ODP, etc.), verify your contract terms. Compliant alternatives are ODP via OData or SLT replication. Full note at support.sap.com/notes/3255746 (SAP login required). [NEEDS_SAP_CITATION — verify current note version]

  2. Released CDS view — confirm the extraction-safe view exists.

    Every table covered in this Academy has a SAP-released CDS view annotated for extraction. Open SE80 and search for the view name before you build anything. Confirm the annotation @Analytics.dataExtract: true is present. If the view is absent or the annotation is missing, stop — raw table extraction is not the solution.

  3. Authorizations — grant the extraction user minimum permissions.

    Your extraction user needs three authorization objects. Use SU01 to create a System-type (non-dialog) user, then use PFCG to assign a role containing:

    • S_RFC — allows RFC calls (set RFC_TYPE=FUNC)
    • S_ODP_READ — allows reading from ODP contexts (set ODPSOURCE=ABAP_CDS for CDS-based extraction)
    • S_TABU_DIS — table display access for the relevant authorization group

    For SLT-based extraction, your Basis team will also need authorization in LTCO and LTRS. [NEEDS_SAP_CITATION — confirm exact auth objects for LTCO/LTRS]

  4. Row-count context — understand data volumes before you run.

    Use SE16N to count rows in the target table with your intended filter (company code, fiscal year, material type, etc.) before you start the extraction. This gives you:

    • A baseline to reconcile against after the extract.
    • A sense of whether your partition is safe to extract (< 100M rows is generally safe for ODP; larger partitions may need further splitting or SLT).
    • Early warning if a filter is misconfigured (count returns billions — rethink the partition).

These four checks take 15–30 minutes on a first walkthrough, and under 5 minutes once you know your system. They are the difference between a smooth extraction and a multi-day incident.

Overview

Scenario

Your finance warehouse needs real-time GL entries (BKPF + BSEG combined). You will use SLT parallel readers and two separate replication objects to stream billions of line items with sub-minute latency.

This walkthrough focuses on the SAP-side work: cardinality analysis, LTCO dual-table configuration, LTRS tuning, and verifying join coherence between BKPF and BSEG once both full loads complete.

What you have built

You have coordinated an enterprise-scale GL fact table fed by real-time SLT replication of BKPF and BSEG. The SAP-side configuration is correct, join coherence is verified, and the delta queue is stable. Finance can now run sub-minute financial close processes on current data.

Before you start

  • Completed BKPF Intermediate walkthrough
  • Understanding of BSEG table structure (accounting line items)
  • Full Use SAP license confirmed
0 of 5 steps complete
  1. Step 1

    Understand BKPF + BSEG cardinality in SE16N

    BKPF (header, 1 row per document) joins to BSEG (line items, N rows per document). For a fiscal year partition, use SE16N to count both tables. The BSEG/BKPF row ratio is typically 8–15×. Understanding this ratio is essential for sizing your target storage and estimating full-load time.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → BKPF → Filter BUKRS='1000' AND GJAHR=2024 → Count; repeat for BSEG.
    Verify
    You have row counts for both BKPF and BSEG for the same BUKRS+GJAHR partition. BSEG/BKPF ratio is documented.
  2. Step 2

    Configure separate SLT replication objects for BKPF and BSEG in LTCO

    In LTCO, create two replication objects: one for BKPF (partition key BUKRS+GJAHR), one for BSEG (partition key BUKRS+GJAHR+BUZEI — adding line item for finer partitioning). Separate objects let you monitor and manage each table independently. This is important because BSEG full load will take significantly longer than BKPF.

    Do this in SAP
    Open transaction LTCO (opens in new tab) and navigate to Replication Objects → Create → BKPF (BUKRS+GJAHR); Create → BSEG (BUKRS+GJAHR+BUZEI).
    Verify
    LTCO shows two active replication objects: BKPF_REP and BSEG_REP. Partition keys are correctly configured.
  3. Step 3

    Configure LTRS parallel readers for BSEG

    BSEG is typically 10× larger than BKPF. Configure more parallel readers for BSEG in LTRS to compensate. For BKPF, 2 readers is usually sufficient. For BSEG, start with 4 and increase to 8 if the full load takes longer than expected without causing SM50 saturation.

    Do this in SAP
    Open transaction LTRS (opens in new tab) and navigate to Administration → Settings → BKPF: 2 readers; BSEG: 4 readers.
    Verify
    LTRS shows correct reader counts per replication object. SM50 confirms work process utilization is within safe limits.
  4. Step 4

    Trigger full replication for both tables and monitor

    Start both full replications from LTCO. Monitor SM50 to ensure combined load from both replications stays safe. BKPF full load completes first; it will switch to DELTA while BSEG is still in full load. This is normal and expected — the two tables replicate independently.

    Do this in SAP
    Open transaction LTCO (opens in new tab) and navigate to Replication Objects → BKPF_REP → Full Replication; BSEG_REP → Full Replication.
    Verify
    Both replication objects show RUNNING status. BKPF completes and shows DELTA; BSEG is still RUNNING. SM50 work process utilization below 80%.
  5. Step 5

    Monitor join coherence — no orphan BSEG rows

    After both full loads complete, verify referential integrity: every BSEG row must have a matching BKPF header. Because the two tables replicate independently, there is a window where a BSEG row exists in your target before the BKPF header arrives. For data quality checks, allow 24 hours after both full loads complete before running orphan queries — the small window closes once DELTA is stable for both tables.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → BSEG → Filter BUKRS+GJAHR → Count. Verify matches BKPF line item total..
    Verify
    SE16N BKPF count × average line items ≈ SE16N BSEG count for the same partition. In your target, after 24 hours: orphan BSEG rows (no matching BKPF) = 0.

Troubleshooting

LTRS parallel readers cause lock contention in SAP

Reduce reader count from 8 to 4 for BSEG. Add a 30-second delay between reader launches in LTRS configuration. This reduces the burst of parallel reads and allows other SAP processes to access BSEG without lock waits.

BSEG target lag is much higher than BKPF

BSEG is 10× larger — a higher lag during full load is expected. Once both tables are in DELTA mode, the lag should equalise. If BSEG delta lag remains high after full load completes, your target sink may be the bottleneck. Investigate target write throughput.
