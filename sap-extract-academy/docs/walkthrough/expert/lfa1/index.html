Extract Vendor Master Tables LFA1 + LFB1 + LFBK (Expert)

SAP-side expert guide for multi-table vendor master extraction. Covers the three-table structure (LFA1 general, LFB1 company code, LFBK bank details), released CDS views, parallel ODP subscription management, and referential integrity verification via SE16N.

Estimated time
100 minutes
Destinations
Generic
Extractors
ODP

Four things to confirm before any tool runs

The checks below, in order, are the difference between a smooth extraction and a multi-day incident. Each one is one sentence; expand for the how.

  1. 01 · Licence

    Confirm your extraction method is covered by your installed SAP licence before anyone designs anything.

    How to check

    Runtime licences cover ODP through OData. SLT replication requires a Full Use licence. Ask your SAP licensing contact; do not assume. See the dedicated article on Runtime vs Full Use, and note that SAP Note 3255746 prohibits third-party tools from using ODP via RFC — compliant paths are ODP over OData or SLT. [NEEDS_SAP_CITATION]

  2. 02 · Released CDS view

    Every table covered here has an extraction-safe released CDS view; confirm it exists before building.

    How to check

    Open the view in SE80 (or in ADT) and confirm the header annotation @Analytics.dataExtract: true is present. If it is missing, the view is not extractable — raw-table reads are not the fix.

  3. 03 · Authorisations

    A System-type technical user with the minimum authorisation objects for your chosen method — no named users, no wildcards.

    How to check

    Create a System user via SU01 and assign a role via PFCG. For ODP: S_RFC (RFC_TYPE=FUNC) and S_ODP_READ (ODPSOURCE=ABAP_CDS). For SLT your Basis team also needs authorisation in LTCO and LTRS. For RFC direct add S_TABU_DIS scoped to the table's authorisation group — never wildcard it.

  4. 04 · Row-count baseline

    Use SE16N to measure the intended partition before extraction — you cannot reconcile without this number.

    How to check

    Open SE16N, apply the same filter your extraction will use, and record the count. Under 100M rows is usually safe for a single ODP partition; larger volumes need further splitting or SLT. A count in the billions means the filter is misconfigured — rethink the partition before running anything.

License check

All licenses — ODP extraction is permitted for all license types. License trap: Runtime vs Full Use →

ODP (Operational Data Provisioning) is permitted under both Runtime and Full Use SAP licenses when you access data through published CDS views via the OData API.

SAP Note 3255746 (Feb 2024): Third-party tools that call ODP via RFC (ADF SAP CDC, Qlik, Fivetran ODP, Informatica ODP, etc.) are no longer permitted. Use ODP via OData, or switch to SLT. Verify with your licensing contact. Full note (SAP login required). [NEEDS_SAP_CITATION — confirm current note state]

SLT (SAP Landscape Transformation) requires a Full Use SAP license. It is not permitted under Runtime licenses. Verify in SLICENSE that "SLT Landscape Transformation" is listed as Active, Type = Full Use before you configure anything in LTCO.

License trap: Runtime vs Full Use →

Direct RFC extraction (e.g., RFC_READ_TABLE) is generally permitted for read-only access under Runtime and Full Use licenses, but is not recommended for production use — it bypasses authorization checks that released CDS views enforce and is subject to SAP Note 3255746 restrictions when wrapped by third-party ODP connectors. Overview

Scenario

Your data warehouse needs a complete vendor dimension: general data (LFA1) + company code details (LFB1) + bank information (LFBK). You will extract all three tables in parallel and verify referential integrity.

This walkthrough covers the SAP-side work: identifying the released CDS views for all three tables, monitoring three parallel ODQMON subscriptions, and verifying join coherence after extraction.

What you have built

You have a robust, multi-table vendor extraction with confirmed referential integrity. All three vendor tables are correctly extracted and reconciled against SE16N baselines. The downstream vendor dimension has complete data for all company code and bank detail combinations.

Before you start

  • Completed LFA1 Intermediate walkthrough
  • Understanding of vendor master table relationships (1:N cardinality)
0 of 5 steps complete
  1. Step 1

    Understand vendor table structure and relationships

    LFA1 (vendor general data) links to LFB1 (vendor company code data) via LIFNR. LFB1 links to LFBK (vendor bank details) via LIFNR+BANKS+BANKL. A single vendor may have records in all three tables. Use SE11 to inspect the foreign key relationships on each table and confirm the join conditions before building any extraction.

    Do this in SAP
    Open transaction SE11 (opens in new tab) and navigate to Database Tables → LFB1 → Foreign Keys.
    Verify
    SE11 confirms: LFB1.LIFNR → LFA1.LIFNR (1:N), LFBK.LIFNR → LFA1.LIFNR (1:N). You understand the join conditions.
  2. Step 2

    Confirm released CDS views for all three tables

    Verify the three released CDS views exist in SE80: I_Supplier (LFA1), I_SupplierCompanyCode (LFB1), I_SupplierBankDetail (LFBK — may also appear as I_SupplierBank depending on your release). [NEEDS_SAP_CITATION — confirm view names for LFB1 and LFBK across releases] Each must have @Analytics.dataExtract: true.

    Do this in SAP
    Open transaction SE80 (opens in new tab) and navigate to Repository Browser → Search → I_Supplier, I_SupplierCompanyCode, I_SupplierBankDetail.
    Verify
    All three views exist with @Analytics.dataExtract: true. View names match your system's release.
  3. Step 3

    Count rows in all three tables (SE16N baseline)

    Use SE16N to count LFA1, LFB1, and LFBK separately. Record each count. The LFB1 count will be higher than LFA1 (a vendor can have company code records in multiple company codes). LFBK will typically be lower than LFA1 (not all vendors have bank details).

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → LFA1, LFB1, LFBK → Count each.
    Verify
    Three row counts recorded. LFB1 >= LFA1 (expected). LFBK <= LFA1 (expected).
  4. Step 4

    Monitor three parallel ODP subscriptions in ODQMON

    When your tool extracts all three tables in parallel, ODQMON shows three separate subscriptions. Monitor all three for status. A failure in LFB1 or LFBK while LFA1 succeeds is a common pattern — retry failed table extractions independently. Create separate monitoring alerts for each subscription.

    Do this in SAP
    Open transaction ODQMON (opens in new tab) and navigate to Subscriptions → Context: ABAP_CDS → I_Supplier, I_SupplierCompanyCode, I_SupplierBankDetail.
    Verify
    All three subscriptions are active and progressing. No subscription in ERROR state.
  5. Step 5

    Verify referential integrity after extraction

    After all three tables extract, verify that every LFB1.LIFNR exists in LFA1 and every LFBK.LIFNR exists in LFA1. Small referential integrity gaps can occur if a vendor was created in LFB1 but the LFA1 record arrived in a different extraction window. Allow a 24-hour delta window before declaring the integrity check a failure — independent table extractions will have small timing gaps.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Use SE16N to cross-check LIFNR values across all three tables.
    Verify
    After 24 hours: all LFB1 and LFBK rows have matching LIFNR in LFA1. Any orphan rows are investigated and resolved.

Troubleshooting

LFB1 or LFBK extractions fail while LFA1 succeeds

Retry failed table extractions independently. Each ODP subscription is stateful and independent. ODQMON shows the specific error for the failed subscription. Check that the extraction user's S_ODP_READ authorization covers all three CDS views (I_Supplier, I_SupplierCompanyCode, I_SupplierBankDetail).

LFB1 count in target is significantly lower than SE16N

I_SupplierCompanyCode may apply a filter to specific company codes your user is authorized for (via S_TABU_DIS or a company code authorization object). Verify the extraction user has authorization for all company codes in scope, or document that the extraction is scoped to authorized company codes only.
