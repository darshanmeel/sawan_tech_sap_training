Extract ACDOCA Enterprise Scale (Expert)

SAP-side expert guide for enterprise-scale ACDOCA extraction via SLT. Covers license verification in SLICENSE, SLT replication object configuration in LTCO, parallel reader setup in LTRS, SM50 load monitoring, and ODQMON delta reconciliation. Full Use license is mandatory.

Four things to confirm before any tool runs

The checks below, in order, are the difference between a smooth extraction and a multi-day incident. Each one is one sentence; expand for the how.

  1. 01 · Licence

    Confirm your extraction method is covered by your installed SAP licence before anyone designs anything.

    How to check

    Runtime licences cover ODP through OData. SLT replication requires a Full Use licence. Ask your SAP licensing contact; do not assume. See the dedicated article on Runtime vs Full Use, and note that SAP Note 3255746 prohibits third-party tools from using ODP via RFC — compliant paths are ODP over OData or SLT. [NEEDS_SAP_CITATION]

  2. 02 · Released CDS view

    Every table covered here has an extraction-safe released CDS view; confirm it exists before building.

    How to check

    Open the view in SE80 (or in ADT) and confirm the header annotation @Analytics.dataExtract: true is present. If it is missing, the view is not extractable — raw-table reads are not the fix.

  3. 03 · Authorisations

    A System-type technical user with the minimum authorisation objects for your chosen method — no named users, no wildcards.

    How to check

    Create a System user via SU01 and assign a role via PFCG. For ODP: S_RFC (RFC_TYPE=FUNC) and S_ODP_READ (ODPSOURCE=ABAP_CDS). For SLT your Basis team also needs authorisation in LTCO and LTRS. For RFC direct add S_TABU_DIS scoped to the table's authorisation group — never wildcard it.

  4. 04 · Row-count baseline

    Use SE16N to measure the intended partition before extraction — you cannot reconcile without this number.

    How to check

    Open SE16N, apply the same filter your extraction will use, and record the count. Under 100M rows is usually safe for a single ODP partition; larger volumes need further splitting or SLT. A count in the billions means the filter is misconfigured — rethink the partition before running anything.

License check

ODP (Operational Data Provisioning) is permitted under both Runtime and Full Use SAP licenses when you access data through published CDS views via the OData API.

Verify with your licensing contact. Full note (SAP login required). [NEEDS_SAP_CITATION — confirm current note state]

SLT (SAP Landscape Transformation) requires a Full Use SAP license. It is not permitted under Runtime licenses. Verify in SLICENSE that "SLT Landscape Transformation" is listed as Active, Type = Full Use before you configure anything in LTCO.

Direct RFC extraction (e.g., RFC_READ_TABLE) is generally permitted for read-only access under Runtime and Full Use licenses, but is not recommended for production use — it bypasses authorization checks that released CDS views enforce and is subject to SAP Note 3255746 restrictions when wrapped by third-party ODP connectors. Overview

Scenario

You are the data architect for a large enterprise. The CFO needs real-time GL data in the data warehouse: every posting visible within minutes, all historical ACDOCA, correct currency handling, and airtight reconciliation.

This is the most demanding SAP extraction scenario. This walkthrough focuses on the SAP-side configuration work: the license check, SLT setup, parallel reader configuration, system load monitoring, and post-load reconciliation. Your infrastructure team owns the target sink configuration (Kafka, S3, ADLS, or equivalent); your Basis team owns LTCO and LTRS.

Your role is to specify the correct partition strategy, ensure the license is in place, and verify the extraction is complete and accurate.

What you have built

You have coordinated an enterprise-scale, real-time GL extraction from SAP. Every posting from every company in every fiscal year flows to your target within minutes of posting. LTCO and LTRS are configured with the correct partitioning. SM50 confirms SAP is not overloaded. ODQMON and SE16N confirm the data is complete.

The patterns here — license-first, partition-before-running, parallel readers, continuous monitoring — apply to every large table. BKPF, EKPO, VBRK, LIPS all follow the same architecture at scale.

  1. Step 1

    License check — verify Full Use in SLICENSE

    SLT replication of ACDOCA requires Full Use License. SAP does not enforce this at the technical level — the transaction will run regardless. But an audit will detect it and the consequences are severe. Open SLICENSE and verify that "SLT Landscape Transformation" is listed as Active, Type = Full Use. Get this confirmed in writing from your SAP licensing contact before proceeding.

    Do this in SAP
    Open transaction SLICENSE (opens in new tab) and navigate to License → License Overview → Check SLT entry.
    Verify
    SLICENSE shows SLT Landscape Transformation = Active, Type = Full Use. Written confirmation from SAP licensing exists.
    Why this matters

    Proceeding without confirmed Full Use license results in licensing violations and potential audit findings with retroactive costs. SAP audits SLT usage. Do not rely on the transaction running as proof of license compliance.

  2. Step 2

    Confirm ACDOCA partition sizes before configuring SLT

    Before configuring SLT, use SE16N to count ACDOCA rows per company code (RBUKRS) and fiscal year (RYEAR). This determines how many SLT parallel readers you need and how long the full load will take. At 4 parallel readers, a 2B-row ACDOCA typically takes 48–72 hours for the initial full load.

    Do this in SAP
    Open transaction SE16N (opens in new tab) and navigate to Execute → ACDOCA → Filter each RBUKRS and RYEAR combination.
    Verify
    You have a per-(RBUKRS, RYEAR) row count. Total row count and number of distinct company code + fiscal year combinations are documented.
  3. Step 3

    Configure SLT replication object in LTCO

    In LTCO, create a replication object for ACDOCA. Set the partitioning key to RBUKRS + RYEAR — this is mandatory for ACDOCA at scale. Without partitioning, SLT readers will attempt to read the entire table in a single pass and cause memory errors. Your Basis team owns the LTCO configuration; your role is to specify the correct partition key and confirm the replication object is set up correctly.

    Do this in SAP
    Open transaction LTCO (opens in new tab) and navigate to Replication Objects → Create → Source Table: ACDOCA → Partition Key: RBUKRS+RYEAR.
    Verify
    LTCO shows an ACDOCA replication object with partition key RBUKRS+RYEAR. Your Basis team confirms the SLT cluster is connected and the replication object is in a ready state.
  4. Step 4

    Configure LTRS parallel readers (4–8 readers)

    In LTRS, configure the number of parallel readers. Each reader handles one partition independently. For ACDOCA, 4 readers is the minimum recommended; 8 is the maximum before diminishing returns (and risk of exhausting SAP work processes). Coordinate with Basis on the appropriate value for your system size. [NEEDS_SAP_CITATION — confirm recommended LTRS reader counts for large tables]

    Do this in SAP
    Open transaction LTRS (opens in new tab) and navigate to Administration → Settings → Parallel Readers → Set to 4.
    Verify
    LTRS shows 4 reader processes configured. Basis team confirms the SLT cluster has enough resources to run 4 readers without starving productive users.
  5. Step 5

    Monitor SAP system load during full replication (SM50)

    ACDOCA full replication runs for 48–72 hours and stresses SAP. Monitor SM50 throughout. SLT processes appear as wp_SLT* or background work processes. Total work process utilization must stay below 80% to avoid impacting finance posting users. Coordinate the full-load timing with your finance team — start during a low-activity window.

    Do this in SAP
    Open transaction SM50 (opens in new tab) and navigate to Work Process Overview → Filter SLT processes.
    Verify
    SM50 shows SLT processes active during replication. Total work process utilization is below 80%. Finance users are not reporting posting failures.
  6. Step 6

    Trigger and monitor full replication

    In LTCO, trigger the full replication. Monitor progress via LTCO status screen. The status moves through INITIALIZING → RUNNING → COMPLETED (for full load), then automatically switches to DELTA for ongoing change capture. Do not stop or restart the full load unless it errors — an interrupted full load may leave partial data in your target and require a complete restart.

    Do this in SAP
    Open transaction LTCO (opens in new tab) and navigate to Replication Objects → ACDOCA_REP → Full Replication.
    Verify
    LTCO status = RUNNING. Progress (% rows replicated) is visible and advancing. Estimate completion time based on current rows/second rate.
  7. Step 7

    Confirm switch to real-time delta after full load

    After the full load completes, LTCO automatically switches to DELTA mode. Check LTRS to confirm the delta queue is flowing and lag is near zero. Real-time posting updates should now flow to your target within seconds to minutes, depending on SLT configuration and network latency. [NEEDS_SAP_CITATION — confirm typical SLT delta latency figures]

    Do this in SAP
    Open transaction LTRS (opens in new tab) and navigate to Monitor → Replication Status → DELTA.
    Verify
    LTCO status = DELTA. LTRS shows delta queue depth near zero (no backlog). New postings in SAP appear in your target within the expected latency window.
  8. Step 8

    Reconcile ACDOCA totals in ODQMON and SE16N

    After full load, reconcile: (1) Use ODQMON to confirm queue depth is near zero and no subscription errors exist. (2) Use SE16N to count ACDOCA rows per (RBUKRS, RYEAR) and compare to your target system. Allow for in-flight postings (rows added during the extraction window).

    Do this in SAP
    Open transaction ODQMON (opens in new tab) and navigate to Subscriptions → ACDOCA delta status.
    Verify
    ODQMON shows no error states. SE16N per-partition counts match target counts within a tolerable delta for in-flight postings. Document this as the post-load reconciliation record.

Troubleshooting

SLT replication crashes after 12 hours with memory error in SAP

Reduce parallel reader count from 8 to 4 in LTRS. Restart the full load. The extension in duration is worth the gain in stability. Also confirm ACDOCA partition key is correctly set — memory errors during full load are often caused by a missing partition key forcing a single-reader full-table scan.

LTCO shows DELTA status but lag is growing continuously

The SLT process is not keeping up with the posting rate. Check SM50 to confirm SLT processes are running. Potential causes: network bottleneck between SLT and target, resource contention on the SAP system, or the target sink is the bottleneck. Investigate each layer in sequence.

License violation notice from SAP

You proceeded without Full Use license. Contact SAP licensing immediately. Do not decommission SLT before consulting with SAP — unilateral decommissioning after an audit finding can complicate the resolution. Engage your SAP account team.
