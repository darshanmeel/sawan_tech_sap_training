expert ACDOCA

Extract ACDOCA Enterprise Scale (Expert)

Full-enterprise ACDOCA extraction: multiple companies, multiple years, billions of rows. SLT push replication to Kafka with log compaction. Partitioning by BUKRS + GJAHR. The ultimate SAP extraction challenge and this site's flagship walkthrough.

Estimated time
150 minutes
Destinations
Kafka
Extractors
SLT

License check

MANDATORY: Full Use SAP License. SLT replication is NOT permitted under Runtime License. Verify licensing with SAP before proceeding.

Overview

Scenario

You are the data architect for a multi-billion-dollar enterprise. The CFO needs real-time GL data in the data warehouse: every posting visible within minutes, all historical ACDOCA (decades of data), correct currency handling, and airtight reconciliation to SAP's GL.

This is the ultimate SAP extraction challenge. You have:

  • A properly-licensed S/4HANA system (Full Use)
  • A Kafka cluster ready for streaming data
  • Snowflake warehouse ready for analytics
  • 2-3 days to build the pipeline
  • Zero tolerance for errors (finance data)

This walkthrough is the culmination of everything you've learned.

What you've built

You have built an enterprise-scale, real-time GL pipeline that streams billions of accounting entries from SAP to your data lake. Every posting from every company in every fiscal year is available in Snowflake within minutes.

ACDOCA at scale is the hardest extraction problem in SAP. You've solved it using the industry-standard approach: SLT partitioning, LTRS parallelism, Kafka queueing, and Snowflake upserts.

The patterns here — partitioning, parallel readers, cardinality reconciliation, Z-field transformations — apply to every large table. BKPF, EKPO, VBRK, LIPS, MSKU — they all follow the same architecture.

You are now ready to extract any SAP table at any scale.

Post-Extraction

Once this pipeline is live, your work shifts to operational excellence:

  1. Monitoring — Dashboard for replication lag, Kafka topic growth, Snowflake ingestion rate
  2. Alerting — Slack notifications if lag > 5 min, Kafka space < 50GB free
  3. Reconciliation — Daily GL reconciliation job comparing SAP aggregates to Snowflake
  4. Documentation — Runbook for on-call support (how to restart SLT, troubleshoot lag, etc.)
  5. Performance tuning — Once stable, optimize Snowflake materialized views for BI queries

Good luck. You've earned it.

Before you start

  • Completed ACDOCA Intermediate
  • Completed BKPF Expert
  • SLT licensed and configured (Full Use)
  • Kafka cluster (10+ brokers, 500GB+ storage)
0 of 10 steps complete
  1. Step 1

    License check: Full Use or fail fast

    SLT replication of ACDOCA requires Full Use License. Runtime Licenses are rejected by SAP. Check your license certificate in SLICENSE before proceeding. This is non-negotiable.

    Do this in SAP
    Open transaction SLICENSE (opens in new tab) and navigate to License → License Overview → Check SLT entry.
    Verify
    License shows 'SLT Landscape Transformation' = Active, Type = Full Use
    Why this matters

    Proceeding without correct license will result in licensing violations and potential audit findings. SAP audits SLT usage rigorously.

  2. Step 2

    Right-size Kafka infrastructure

    ACDOCA can be 500B+ rows annually. Kafka cluster needs adequate disk space, replication factor 3, and log compaction enabled. Budget 10+ brokers, 500GB minimum.

    Verify
    Topic acdoca-source exists with 30 partitions.
  3. Step 3

    Configure SLT to partition by BUKRS + GJAHR

    In LTCO, define replication object for ACDOCA with partitioning: BUKRS + GJAHR. This avoids hot partitions and enables parallel SLT readers.

    Do this in SAP
    Open transaction LTCO (opens in new tab) and navigate to Replication Objects → ACDOCA → Partitioning.
    Verify
    LTCO shows ACDOCA_KAFKA replication object with partition key BUKRS+GJAHR.
  4. Step 4

    Enable LTRS parallel readers (4-8 readers)

    SLT's Landscape Transformation Replication Server (LTRS) can run 4-8 parallel database readers. Each reader processes a company code + fiscal year partition independently.

    Do this in SAP
    Open transaction LTRS (opens in new tab) and navigate to Administration → Settings → Parallel Readers.
    Verify
    LTRS shows 4 reader processes active and running.
  5. Step 5

    Monitor SAP load during full replication

    ACDOCA full replication will stress SAP for days. Monitor CPU, memory, lock wait times via SM50. Coordinate with SAP Basis to ensure no conflicts with batch jobs.

    Do this in SAP
    Open transaction SM50 (opens in new tab) and navigate to Monitor → Active Sessions.
    Verify
    SM50 shows SLT processes (wp_SLT*) running, CPU usage acceptable (< 80%).
  6. Step 6

    Trigger full load

    In LTCO, trigger full replication. This will run for 24-72 hours depending on ACDOCA size. Kafka topic will fill with billions of rows.

    Do this in SAP
    Open transaction LTCO (opens in new tab) and navigate to Replication Objects → ACDOCA_KAFKA → Full Replication.
    Verify
    Replication status = 'RUNNING'. Monitor Kafka topic growth via kafka-log-dirs tool.
  7. Step 7

    Switch to real-time delta

    Once full load completes, SLT automatically captures delta. Real-time posting updates flow to Kafka within seconds.

    Verify
    LTCO status = 'DELTA'. LTRS shows delta queue depth (should be near zero if no backlog).
  8. Step 8

    Consume from Kafka with Snowflake Connector

    Snowflake Kafka Connector reads acdoca-source topic, applies Z-field transformations (CUKY pairing for currencies), upserts into Snowflake GL_ACDOCA_RAW.

    Verify
    Connector status = RUNNING. Snowflake GL_ACDOCA_RAW table grows.
  9. Step 9

    Reconcile amounts and cardinality

    Compare aggregate amounts in ACDOCA (SAP) vs GL_ACDOCA_RAW (Snowflake) for each company + year. Cardinality should match exactly. Monitor for 24 hours to ensure delta is stable.

    Verify
    Totals match exactly. No orphan or duplicate rows.
  10. Step 10

    Document Z-field transformations

    ACDOCA contains CUKY (currency key) and related amount fields (AMOUNT_TRANS in transaction currency vs AMOUNT_LOCAL in local currency). Document the transformation logic in your data dictionary.

    Verify
    Data dictionary includes: ACDOCA.AMOUNT_TRANS → GL_ACDOCA_RAW.AMOUNT_TXN_CURRENCY, with conversion rules.

Troubleshooting

SLT replication crashes after 12 hours (memory error in SAP)

Reduce parallel reader count from 8 to 4. Restart full load. This may extend duration but increases stability.

Kafka topic lag grows unbounded (Snowflake connector can't keep up)

ACDOCA is faster than Snowflake can ingest. Scale Snowflake compute (increase warehouse size) or add a second consumer instance.

License violation notice from SAP

You proceeded without Full Use License. Contact SAP immediately. Undo SLT configuration or purchase Full Use License upgrade.
