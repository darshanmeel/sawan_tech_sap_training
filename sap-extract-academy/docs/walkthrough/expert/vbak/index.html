Overview

Scenario

Your analytics platform needs VBAK updates in near-real-time. SLT push replication to your target is the approach: it captures every insert, update, and delete at the database level and delivers them faster than any application-level extraction.

This walkthrough covers SAP-side configuration: license check, RFC destination validation, LTCO replication object setup, and LTRS lag monitoring. Your infrastructure team owns the target sink; your Basis team owns LTCO and LTRS configuration.

What you have built

You have coordinated an enterprise-scale, real-time VBAK replication from SAP. SLT replicates changes within the target SLA, and you have confirmed SM59, LTCO, and LTRS are all healthy. This architecture extends to 10+ tables using the same SAP-side patterns.

Before you start

License check

SLT replication of VBAK requires Full Use SAP License. This is NOT permitted under Runtime License. License trap: Runtime vs Full Use →

ODP (Operational Data Provisioning) is permitted under both Runtime and Full Use SAP licenses when you access data through published CDS views via the OData API.

SAP Note 3255746 (Feb 2024): Third-party tools that call ODP via RFC (ADF SAP CDC, Qlik, Fivetran ODP, Informatica ODP, etc.) are no longer permitted. Use ODP via OData, or switch to SLT. Verify with your licensing contact. Full note (SAP login required).

SLT (SAP Landscape Transformation) requires a Full Use SAP license. It is not permitted under Runtime licenses. Verify in SLICENSE that "SLT Landscape Transformation" is listed as Active, Type = Full Use before you configure anything in LTCO.

License trap: Runtime vs Full Use →

Direct RFC extraction (e.g., RFC_READ_TABLE) is generally permitted for read-only access under Runtime and Full Use licenses, but is not recommended for production use — it bypasses authorization checks that released CDS views enforce and is subject to SAP Note 3255746 restrictions when wrapped by third-party ODP connectors.

SAP prerequisites

These steps are one-time per SAP system (not repeated per table). Once your RFC user, authorizations, and RFC destination are set up, every table extraction reuses the same connection. The checks below, in order, are the difference between a smooth extraction and a multi-day incident.

  1. 01 · Licence

    Confirm your extraction method is covered by your installed SAP licence before anyone designs anything.

    How to check

    Runtime licences cover ODP through OData. SLT replication requires a Full Use licence. Ask your SAP licensing contact; do not assume. See the dedicated article on Runtime vs Full Use, and note that SAP Note 3255746 prohibits third-party tools from using ODP via RFC — compliant paths are ODP over OData or SLT.

  2. 02 · Released CDS view

    Every table covered here has an extraction-safe released CDS view; confirm it exists before building.

    How to check

    Open the view in SE80 (or in ADT) and confirm the header annotation @Analytics.dataExtract: true is present. If it is missing, the view is not extractable — raw-table reads are not the fix.

  3. 03 · Authorisations

    A System-type technical user with the minimum authorisation objects for your chosen method — no named users, no wildcards.

    How to check

    Create a System user via SU01 and assign a role via PFCG. For ODP: S_RFC (RFC_TYPE=FUNC) and S_ODP_READ (ODPSOURCE=ABAP_CDS). For SLT your Basis team also needs authorisation in LTCO and LTRS. For RFC direct add S_TABU_DIS scoped to the table's authorisation group — never wildcard it.

  4. 04 · Row-count baseline

    Use SE16N to measure the intended partition before extraction — you cannot reconcile without this number.

    How to check

    Open SE16N, apply the same filter your extraction will use, and record the count. Under 100M rows is usually safe for a single ODP partition; larger volumes need further splitting or SLT. A count in the billions means the filter is misconfigured — rethink the partition before running anything.

Also confirm

  • Full Use SAP license confirmed in writing
  • Basis team available for SLT configuration
  • Technical extraction user (System type) created in SU01 with RFC access to the SAP system

SAP side

Steps to complete in your SAP system. Use the Method picker above to switch between ODP, SLT, and RFC approaches.

0 of 6 steps complete
  1. Step 1

    Verify Full Use License in SLICENSE

    Open SLICENSE and confirm SLT Landscape Transformation is listed as Active, Type = Full Use. Get written confirmation from your SAP licensing contact. SLT will run on any system regardless of license — SAP enforces compliance through audits, not technical blocks.

    Do this in SAP
    Open transaction SLICENSE (opens in new tab) and navigate to License Overview → Check SLT entry.
    Verify
    SLICENSE shows SLT Landscape Transformation = Active, Full Use. Written licensing confirmation exists.
  2. Step 2

    Validate the RFC destination for the target system (SM59)

    SLT pushes data to a target system via an RFC destination. Confirm the RFC destination your Basis team has configured points to the correct target and the connection test passes. Use SM59 to run the test. A failed RFC destination will silently drop replication data without SLT raising an error in LTCO.

    Do this in SAP
    Open transaction SM59 (opens in new tab) and navigate to RFC Connections → TCP/IP → Test Connection.
    Verify
    Test Connection returns green for the RFC destination your SLT replication object uses.
  3. Step 3

    Configure SLT replication object for VBAK in LTCO

    Work with your Basis team to create a replication object for VBAK in LTCO. For VBAK, partition by VKORG (sales organisation) to distribute load evenly across SLT readers. Confirm the replication object is in ACTIVE status before triggering any load.

    Do this in SAP
    Open transaction LTCO (opens in new tab) and navigate to Replication Objects → Create → Source Table: VBAK → Partition Key: VKORG.
    Verify
    LTCO shows a VBAK replication object with VKORG as partition key. Status = ACTIVE.
  4. Step 4

    Perform initial full load via LTCO

    Trigger the full replication in LTCO. SLT reads all VBAK rows and pushes them to the target. Depending on VBAK size, this takes minutes to hours. Monitor progress via LTCO status and confirm SM50 work process utilization stays safe.

    Do this in SAP
    Open transaction LTCO (opens in new tab) and navigate to Replication Objects → VBAK_REP → Full Replication.
    Verify
    Full load completes. LTCO status switches from RUNNING to DELTA automatically.
  5. Step 5

    Confirm real-time delta and monitor lag in LTRS

    After full load, SLT captures changes from VBAK in near-real-time. Monitor lag in LTRS. Expected lag for VBAK is under 60 seconds. If lag grows continuously, investigate SM50 for SLT process saturation or network issues between SAP and the target.

    Do this in SAP
    Open transaction LTRS (opens in new tab) and navigate to Monitor → Replication Status → Lag.
    Verify
    LTRS shows DELTA status and replication lag within SLA (e.g., under 60 seconds for VBAK).
  6. Step 6

    Handle Z-fields at scale

    If VBAK has multiple Z-fields (more than 5), SLT may need a field-mapping configuration to include them in the replication object. Confirm with your Basis team that all Z-fields are included in the LTCO replication object definition. If a Z-field was added after the initial replication object was created, it may need to be added manually and a partial re-init triggered. [NEEDS_SAP_CITATION — confirm SLT behavior when Z-fields are added post-configuration]

    Do this in SAP
    Open transaction LTCO (opens in new tab) and navigate to Replication Objects → VBAK_REP → Field Mapping.
    Verify
    LTCO field mapping includes all VBAK Z-fields. Target system records include Z-field values.
+ Handling custom Z columns

ODP — Extending a CDS view with ABAP Developer

Use EXTEND VIEW ENTITY in ABAP Development Tools (Eclipse/ADT) to add Z columns without modifying the original view. The base view must carry the header annotation @AbapCatalog.viewEnhancementCategory: [#PROJECTION_LIST] — if the owner restricted data sources, only tables listed under @AbapCatalog.extensibility.dataSources can be joined.

  1. Open ADT (Eclipse). In the Project Explorer, locate the data definition of the CDS entity you want to extend.
  2. Right-click the data definition → New Data Definition. Choose your package, enter a name (e.g., Z_EXT_MY_VIEW), and add a description.
  3. Click Next, select your transport request, then choose the Extend View Entity template.
  4. Define your custom Z fields in the element list — pull from the underlying table alias or define a new association.
  5. Activate the view extension (Ctrl + F3). The base view now includes your custom elements.
extend view entity C_BaseStandardView with Z_Ext_My_View
{
  BaseTable.zz_custom_field    as ZZCustomField,
  _NewZTableAssoc
}

Because the ODP framework generates a TransientProvider at runtime from CDS metadata, Z columns added via EXTEND VIEW ENTITY automatically appear in the ODP delta queue. Refresh the schema in your tool (ADF, Databricks) to see new fields. Test locally with ABAP program RODPS_REPL_TEST.

SLT — Adding Z columns via LTRS Rule Assignment

In SLT you add and populate Z columns through Advanced Replication Settings in transaction LTRS.

  1. In your SLT system, run transaction LTRS.
  2. Under Table Settings, you can define the target table structure. To add a Z column that doesn't exist in the source, append it to the target structure here.
  3. Go to the Rule Assignment section → right-click → Add Table to add your replication table.
  4. Click the Create icon and choose either a Field-Related Rule (to populate a specific field) or an Event-Related Rule (to manipulate the whole record).
  5. Write ABAP in the Line of Code section to populate your Z column. For event-related rules use <wa_s_table>-columnname. For field-related rules use the E_ prefix: E_columnname.

RFC — Exposing Z columns via SE11 Append Structure

When extracting via RFC, Z columns added to a SAP transparent table through an Append Structure in transaction SE11 are immediately available — the RFC functions query the ABAP data dictionary dynamically.

  1. In SE11, open the transparent table, go to Extras → Append Structure, and add your Z field. Activate — the underlying database table is updated automatically.
  2. Standard RFC functions (RFC_READ_TABLE, /SAPDS/RFC_READ_TABLE2) will return the column dynamically.
  3. Explicitly request the new Z column in the FIELDS parameter of your RFC call — it is not returned unless listed there.
  4. If using a custom Function Module, update the Export table structure in the FM to include the new columns — it does not auto-discover.

Tool configuration

Custom (Python / pyrfc)

  1. 1. Install dependencies

    Install pyrfc (SAP RFC bindings) and boto3 (AWS) or the appropriate sink library. pyrfc requires the SAP NW RFC SDK.

    pip install pyrfc boto3 pandas

# pyrfc also requires the SAP NetWeaver RFC SDK (download from SAP Service Marketplace)
# export SAPNWRFC_HOME=/path/to/nwrfcsdk   (Linux/Mac)
# set SAPNWRFC_HOME=C:\nwrfcsdk            (Windows)

    Verify: python -c 'import pyrfc; import boto3; print("OK")' prints OK with no errors.

  2. 2. Connect to SAP via RFC and extract VBAK

    Use pyrfc to call the ODP extraction function module. Replace connection parameters with your SAP system details and the EXTRACT_VBAK user credentials.

    import pyrfc
import pandas as pd

conn = pyrfc.Connection(
    ashost='sap-prod.company.com',
    sysnr='00', client='100',
    user='EXTRACT_VBAK',
    passwd='<password>'
)

result = conn.call(
    'RODPS_REPL_ODP_EXTRACT',
    SUBSCRIBER_NAME='PYTHON_VBAK_FULL',
    SUBSCRIBER_TYPE='2',
    SUBSCRIBER_PROCESS='FULL_LOAD',
    DATA_AREA='ABAP_CDS',
    ENTITY_NAME='I_SalesDocument',
    MAX_RECORDS=500000,
    FIELDS=[{'FIELDNAME': '*'}]
)

records = result.get('DATA', [])
df = pd.DataFrame(records)
print(f"Extracted {len(df)} rows from VBAK (I_SalesDocument)")

    Verify: Script prints row count. Compare against SE16N count from SAP-side step 02.

  3. 3. Write to S3 or your sink

    After extracting into a DataFrame, write to your landing zone. Swap the boto3 block for ADLS, GCS, or Snowflake as needed.

    import boto3
from io import StringIO

s3 = boto3.client('s3', region_name='eu-west-1')
csv_buffer = StringIO()
df.to_csv(csv_buffer, index=False)
s3.put_object(
    Bucket='my-sap-landing-bucket',
    Key='sap/vbak/vbak_full_load.csv',
    Body=csv_buffer.getvalue()
)
print(f"Loaded {len(df)} rows to s3://my-sap-landing-bucket/sap/vbak/")

    Verify: File appears in sink. Row count matches SE16N. Column names match I_SalesDocument field names.

  3. 3. Create a Sink Dataset (ADLS / Blob / Synapse)

Databricks (PySpark / SAP connector)

  1. 1. Install pyrfc on your Databricks cluster

    pyrfc requires the SAP NetWeaver RFC SDK. Upload the SDK to DBFS or a volume, then install pyrfc on the cluster. Alternatively, use the Databricks SAP CDC connector if available in your workspace.

    # Install pyrfc on the cluster (init script or cluster library)
pip install pyrfc pandas

# Mount SAP NW RFC SDK to DBFS first:
# dbutils.fs.cp("dbfs:/mnt/tools/nwrfcsdk.tar.gz", "/tmp/nwrfcsdk.tar.gz")

    Verify: Cluster restarts with pyrfc importable. Confirm with: python -c 'import pyrfc'.

  2. 2. Extract VBAK and write to Delta table

    Extract via pyrfc (same as Custom Python variant) and write to a Delta table in Databricks Unity Catalog or DBFS.

    import pyrfc, pandas as pd
from pyspark.sql import SparkSession

spark = SparkSession.builder.getOrCreate()

conn = pyrfc.Connection(
    ashost='sap-prod.company.com',
    sysnr='00', client='100',
    user='EXTRACT_VBAK', passwd='<password>'
)
result = conn.call(
    'RODPS_REPL_ODP_EXTRACT',
    SUBSCRIBER_NAME='DBX_VBAK_FULL',
    SUBSCRIBER_TYPE='2', SUBSCRIBER_PROCESS='FULL_LOAD',
    DATA_AREA='ABAP_CDS', ENTITY_NAME='I_SalesDocument',
    MAX_RECORDS=500000, FIELDS=[{'FIELDNAME': '*'}]
)
df = spark.createDataFrame(pd.DataFrame(result.get('DATA', [])))
df.write.format('delta').mode('overwrite').saveAsTable('sap_landing.raw.vbak')

    Verify: Delta table sap_landing.raw.vbak exists. SELECT COUNT(*) matches SE16N.

Sink setup

Landing zone configuration for your selected sink. Switch the Sink picker above to see a different destination.

Azure Data Lake Storage Gen2

ADLS is your landing zone — your tool writes Parquet, Delta, or Iceberg files here. Downstream engines (Synapse, Databricks, Power BI) read directly from these files.

Native SAP connector option

If your organisation has an ADF SAP connector license (SAP connector for Azure Data Factory), ADF writes directly to ADLS from SAP — no intermediate tool pipeline code needed. Confirm connector licensing with your Azure team before setting up the manual pipeline below.

Before your first extraction

  • Create an ADLS Gen2 storage account and container (e.g., sap-landing).
  • Register an Entra ID (Azure AD) app. Grant it Storage Blob Data Contributor on the container. Store tenant ID, client ID, and secret in your pipeline's secrets manager.
  • Agree on a partition path convention: sap/vbak/rbukrs=1000/year=2024/. Partition columns must match the filter dimensions used in the SAP extraction.
  1. 1. Confirm the file landed after extraction

    After your tool runs, confirm the Parquet (or Delta/Iceberg) file lands in the container at the correct partition path. Use Azure Storage Explorer or the Azure portal. Prefer Parquet with Snappy compression; use Delta or Iceberg if your tool supports table formats.

    Verify: File visible at sap/vbak/rbukrs=1000/year=2024/ in the container.

  2. 2. Validate row count from Synapse or Databricks

    Run a count query directly against the landed files. Compare to the SE16N count from the SAP-side steps. This is your reconciliation checkpoint before handing data to analytics.

    Verify: SELECT COUNT(*) FROM parquet.`abfss://sap-landing@account.dfs.core.windows.net/sap/vbak/...` matches SE16N.

Amazon S3

S3 is your landing zone — your tool writes Parquet files here. Downstream engines (Athena, Redshift Spectrum, Databricks on AWS) read directly from S3, or you COPY INTO Snowflake/Redshift from here.

Native SAP connector option

If your organisation has AWS AppFlow with SAP OData connector or a licensed SAP Integration Suite adapter, AppFlow can land data directly in S3 without tool pipeline code. Check your AWS and SAP BTP licensing before building the pipeline below.

Before your first extraction

  • Create an S3 bucket in your preferred region. Enable server-side encryption (SSE-S3 minimum) and block all public access.
  • Create an IAM role with s3:PutObject and s3:GetObject on the specific bucket/prefix ARN. Use instance profiles or IRSA — avoid long-lived access keys.
  • Agree on a landing prefix convention: sap/vbak/rbukrs=1000/year=2024/.
  1. 1. Confirm files land and register in Glue

    After extraction, confirm files appear at the S3 prefix. Register the prefix as a Glue table pointing to Parquet files so Athena and Spark can discover it. If the next layer is Snowflake, create an external stage on this bucket path.

    Verify: aws s3 ls s3://your-bucket/sap/vbak/ shows landed files. Glue crawler ran and created the table.

  2. 2. Validate row count in Athena

    Run SELECT COUNT(*) FROM your_glue_table in Athena. Compare to SE16N. Athena scans Parquet efficiently if partition columns match the Hive-style path used for landing.

    Verify: Athena count matches SE16N. Downstream team can query the Glue table without moving files.

Google Cloud Storage (GCS)

GCS is your landing zone — your tool writes Parquet or Iceberg files here. BigQuery reads them directly as an external table or via a load job. No intermediate copy needed.

Native SAP connector option

If your organisation has the Google Cloud BigQuery Connector for SAP (available via SAP BTP / SAP LT Replication Server), it streams data directly from SAP ABAP tables into BigQuery without any intermediate GCS files or pipeline code. Confirm with your SAP Basis and GCP teams whether this connector is licensed before building the file-based pipeline below.

Before your first extraction

  • Create a GCS bucket in your preferred region. Enable uniform bucket-level access and Object Versioning.
  • Create a GCP service account. Grant it Storage Object Admin on the bucket. Store the JSON key in Secret Manager.
  • Set a landing prefix: sap/vbak/rbukrs=1000/year=2024/.
  1. 1. Create BigQuery external table or load job

    After landing files in GCS, create a BigQuery external table (zero-copy, always current) or run a load job (faster queries, persistent storage). For Iceberg landing, BigQuery supports Iceberg tables via GCS BigLake — gives full DML and snapshot history.

    Verify: Table sap_landing.vbak exists in BigQuery. Schema matches SE11.

  2. 2. Validate row count

    Run SELECT COUNT(*) FROM sap_landing.vbak. Compare to SE16N. Partition by posting date or extraction date to keep BigQuery query costs manageable at scale.

    Verify: Count matches SE16N. Partition metadata visible in BigQuery console.

Snowflake

Snowflake reads Parquet files your tool landed in ADLS or S3. Two loading patterns: COPY INTO (batch, scheduled) or Snowpipe (continuous, file-triggered — runs automatically when a file lands, no scheduling needed).

Native SAP connector option — skip the steps below if licensed

If your organisation has a Snowflake Native SAP BDC Connector license, Snowflake connects directly to SAP — no tool pipeline code, no external stage, and no Parquet files needed. The connector extracts and loads natively. If this applies to you, configure the connector in Snowflake Partner Connect with your SAP RFC credentials and skip the COPY INTO / Snowpipe steps below.

Before your first extraction (file-based approach)

  • Create a dedicated database (SAP_LANDING), schema (RAW), and warehouse (SAP_LOAD_WH — X-Small for bulk loads).
  • Create a Snowflake storage integration (not access keys) for your ADLS container or S3 bucket. Create an external stage pointing to the landing path where your tool drops Parquet files.
  • Create the target table: CREATE TABLE SAP_LANDING.RAW.vbak ... with columns matching SE11. Or use COPY INTO with MATCH_BY_COLUMN_NAME to auto-map Parquet columns.
  1. 1. Load from external stage — COPY INTO or Snowpipe

    Batch (COPY INTO): Run COPY INTO SAP_LANDING.RAW.vbak FROM @your_stage/sap/vbak/ PATTERN='.*parquet' FILE_FORMAT=(TYPE=PARQUET). Schedule via a Snowflake task or your orchestrator after each extraction.

    Continuous (Snowpipe): Create a pipe: CREATE PIPE sap_vbak_pipe AS COPY INTO SAP_LANDING.RAW.vbak FROM @your_stage/sap/vbak/ FILE_FORMAT=(TYPE=PARQUET). Enable auto-ingest on the stage — Snowpipe fires automatically when a file lands. No scheduling required; ideal for near-real-time SLT feeds.

    Verify: LIST @your_stage/sap/vbak/ shows landed files. COPY INTO or Snowpipe status shows zero skipped files.

  2. 2. Validate row count after load

    Run SELECT COUNT(*) FROM SAP_LANDING.RAW.vbak. Compare to SE16N row count from the SAP-side steps. Check INFORMATION_SCHEMA.LOAD_HISTORY for skipped or errored files if counts diverge.

    Verify: SELECT COUNT(*) matches SE16N. Zero error rows in load history.

BigQuery

BigQuery reads Parquet or Iceberg files from GCS, or accepts direct writes from Fivetran and ADF. If your tool wrote to GCS as Parquet, create an external table — no copy needed.

Native SAP connector option — skip the steps below if licensed

If your organisation has the Google Cloud BigQuery Connector for SAP (part of SAP LT Replication Server or SAP Integration Suite), it streams data directly from SAP into BigQuery tables — no GCS files, no tool pipeline code, no staging needed. Confirm with your SAP Basis team. If licensed, skip the file-based steps below.

Before your first extraction (file-based approach)

  • Create a BigQuery dataset (e.g., sap_landing) in the same region as your GCS bucket. Cross-region reads from GCS are not supported.
  • Grant your service account BigQuery Data Editor on the dataset and Storage Object Viewer on the GCS bucket (for external tables).
  1. 1. Create external table, Iceberg table, or use direct connector

    Parquet path: CREATE EXTERNAL TABLE sap_landing.vbak OPTIONS (format='PARQUET', uris=['gs://your-bucket/sap/vbak/*']) — zero-copy, always current.
    Iceberg path: Register as a BigLake Iceberg table for full DML support and snapshot history.
    Direct connector (Fivetran/ADF): Writes a native BigQuery table directly — no staging or external table needed.

    Verify: Table sap_landing.vbak exists and schema matches SE11.

  2. 2. Validate row count

    Run SELECT COUNT(*) FROM sap_landing.vbak. Compare to SE16N. Partition the table by posting date or extraction date to keep per-query costs manageable at scale.

    Verify: Count matches SE16N. Partition metadata visible in BigQuery console.

Microsoft Fabric / OneLake

Fabric has three file-based paths: (1) ADF writes to Lakehouse via Copy Activity, (2) Databricks writes Delta/Iceberg to ADLS and Fabric shortcuts it, (3) your tool lands Parquet in ADLS and Fabric shortcuts the folder as an external table.

Native SAP connector option — skip the steps below if licensed

If your organisation has the Microsoft Fabric SAP CDC connector or an SAP integration via Azure Logic Apps with native Fabric write support, data flows directly from SAP into OneLake without any intermediate files or tool pipeline code. Confirm with your Microsoft and SAP teams. If licensed, skip the file-based steps below.

Before your first extraction (file-based approach)

  • Create a Lakehouse in your Fabric workspace (e.g., SAP_Landing). Create a Tables/vbak/ folder for Delta/Iceberg tables, or Files/vbak/ for raw Parquet.
  • Decide your ingestion path before extracting: ADF Copy Activity, Databricks ADLS shortcut, or Parquet ADLS shortcut. Each wires up differently — agree with the analytics team first.
  1. 1. Set up your ingestion path

    ADF pipeline: Copy Activity to Lakehouse sink writes directly — no intermediate files needed.
    Databricks Delta/Iceberg: Create an OneLake shortcut pointing to the ADLS path where Databricks wrote. Fabric auto-registers as a Delta table in the SQL endpoint.
    Parquet from any tool: Create an OneLake shortcut to the ADLS Parquet folder. Register as an external table in the Lakehouse.

    Verify: Data appears in the Lakehouse explorer after a test extraction run.

  2. 2. Validate in SQL endpoint

    Open the Lakehouse SQL endpoint in Fabric. Run SELECT COUNT(*) FROM vbak. Fabric automatically registers Delta tables in the SQL endpoint — Parquet external tables appear under Files.

    Verify: Row count matches SE16N. Table visible in Lakehouse explorer.

Databricks Lakehouse

If your tool was Databricks, the Delta or Iceberg table is already written — querying it IS the sink step. If another tool wrote Parquet to ADLS/S3, register it as an external Delta or Iceberg table in Unity Catalog.

Native SAP connector option — skip the steps below if licensed

If your organisation has the Databricks SAP Connector (SAP LT Replication + Databricks native integration), it writes SAP change data directly to a Delta table in real time — no tool pipeline code or external files needed. Confirm with your SAP Basis and Databricks teams. If licensed, skip the file-based steps below.

Before your first extraction (file-based approach)

  • Ensure Unity Catalog is enabled. Create a catalog (e.g., sap_landing) and schema (raw).
  • If Databricks is your extraction tool: the saveAsTable call in the Tool tab writes the table — no further sink setup needed.
  • If another tool writes Parquet to ADLS/S3: note the exact file path for registering as an external table.
  1. 1. Confirm Delta/Iceberg table or register Parquet

    If Databricks was your tool: Run SHOW TABLES IN sap_landing.raw — the table should already exist.
    If tool wrote Parquet: Register as external: CREATE TABLE sap_landing.raw.vbak USING PARQUET LOCATION 'abfss://...'
    Or convert zero-copy to Delta: CONVERT TO DELTA parquet.`abfss://...` — gives time-travel and ACID transactions on the existing files.

    Verify: SELECT COUNT(*) FROM sap_landing.raw.vbak returns the expected row count.

  2. 2. Validate count and grant downstream access

    Compare the row count to SE16N. Then grant SELECT to the analytics team: GRANT SELECT ON TABLE sap_landing.raw.vbak TO `analytics-group`. Downstream users should not need extraction credentials.

    Verify: Row count matches SE16N. Downstream team can query the table.

Other landing zone

Whatever your sink: it receives Parquet, Delta, or Iceberg files from your tool. Ensure your downstream system can read one of these formats efficiently before landing data in it.

Before your first extraction

  • Confirm your sink can read Parquet (most portable), Delta, or Iceberg. Avoid CSV for tables over 10M rows.
  • Agree on a partitioned path convention with the analytics team: sap/vbak/rbukrs=1000/year=2024/run=TIMESTAMP/.
  • Test a small write to your sink before running a full extraction — confirm schema mapping, encoding, and permissions.
  1. 1. Confirm file lands at the correct path

    After extraction, verify the file appears at the agreed partition path. Check file size is reasonable for the expected row count (Parquet with Snappy is typically 100–300 bytes per row for journal data).

    Verify: File is present and non-zero. Schema is readable by your downstream tool.

  2. 2. Row-count reconciliation is mandatory

    Always compare the row count from your sink against the SE16N count from the SAP-side steps. A mismatch means the extraction dropped rows or the filter scopes differ — investigate before handing data to analytics. Automate this check in your pipeline so it fails the run on divergence.

    Verify: Your pipeline has an automated row-count check that fails the run if counts diverge beyond a tolerance threshold.

Troubleshooting

SLT replication lag exceeds SLA

Add a second SLT replication object scoped to a subset of sales organisations to distribute the load. Scale your target sink infrastructure if it is the bottleneck. Check SM50 to confirm SLT processes are not queued waiting for work processes.

SM59 test connection fails after LTCO is configured

The target system's RFC configuration has changed. The Basis team needs to update the SM59 destination with the correct host, port, and credentials. SLT does not alert you if the RFC destination breaks — it silently drops replication events.
