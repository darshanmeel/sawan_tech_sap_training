Extract VBAK via SLT Push to Kafka (Expert)

Enterprise-scale VBAK extraction using SAP Landscape Transformation (SLT) push replication to Apache Kafka with log compaction. Includes multi-partition strategy and Z-field handling at scale.

Estimated time
120 minutes
Destinations
Kafka
Extractors
SLT

License check

SLT replication of VBAK requires Full Use SAP License. This is NOT permitted under Runtime License.

Overview

Scenario

Your company's real-time analytics platform needs VBAK updates in near-real-time. SLT push replication to Kafka is the industry-standard approach: it captures every change at the database level and delivers rows to Kafka faster than any application-level extraction.

You have SLT licensed (Full Use), a Kafka cluster, and Snowflake ready. This walkthrough covers the end-to-end setup from SLT configuration to Snowflake upserts.

Estimated time: 2 hours setup + tuning.

What you've built

You've built an enterprise-scale, real-time VBAK pipeline. SLT replicates changes within minutes, Kafka provides durable queueing and partitioning, and Snowflake maintains an up-to-date replica. This architecture supports analytics dashboards refreshing every few minutes and can be extended to 10+ tables using the same patterns.

Before you start

  • Completed VBAK Intermediate
  • S/4HANA system with SLT configured (RFC connection to source)
  • Kafka cluster (3+ brokers, min 20GB storage)
  • Full Use SAP License (SLT requires it)
0 of 10 steps complete
  1. Step 1

    Verify SLT is installed and licensed

    SAP Landscape Transformation (SLT) replicates tables at the database level, capturing every insert/update/delete. Confirm license with SAP.

    Do this in SAP
    Open transaction LTCO (opens in new tab) and navigate to Administration → System → License.
    Verify
    Full Use License is visible and SLT is listed as an active module.
  2. Step 2

    Configure SLT Kafka connector

    In the SLT cluster (HANA appliance or separate instance), configure the Kafka RFC destination and login credentials.

    Do this in SAP
    Open transaction SM59 (opens in new tab) and navigate to RFC Connections → Create → TCP/IP.
    Verify
    Test connection returns green for the Kafka RFC destination.
  3. Step 3

    Create Kafka topics for VBAK

    Create a compacted Kafka topic for VBAK: vbak-source-topic, replication=3, log.retention=-1, log.cleanup.policy=compact.

    Verify
    Topic vbak-source-topic exists with 10 partitions and log compaction enabled.
  4. Step 4

    Configure SLT Replication Object for VBAK

    In LTCO, create a new replication object: source table VBAK, target Kafka topic vbak-source-topic. Include all fields, enable key-based delta (to leverage log compaction).

    Do this in SAP
    Open transaction LTCO (opens in new tab) and navigate to Replication Objects → Create.
    Verify
    Replication object VBAK_KAFKA is created with status ACTIVE.
  5. Step 5

    Define partitioning strategy

    To avoid hot partitions, partition VBAK by VKORG (sales org) at the SLT level. This distributes load evenly across Kafka partitions.

    Verify
    In LTCO, the replication object shows VKORG as partition key.
  6. Step 6

    Perform initial full load via SLT

    Trigger a full load: SLT reads all VBAK rows and pushes to Kafka in a single batch. Depending on VBAK size, this may take hours.

    Do this in SAP
    Open transaction LTCO (opens in new tab) and navigate to Replication Objects → VBAK_KAFKA → Full Replication.
    Verify
    Full load completes. Kafka topic vbak-source-topic has partitions filled with VBAK rows.
  7. Step 7

    Enable real-time delta (CDC)

    After full load, SLT automatically captures deltas from the VBAK log. New/changed rows are pushed to Kafka in near-real-time (typically sub-minute latency).

    Verify
    In LTCO, monitor shows 'DELTA' status and a heartbeat/row count in LTRS.
  8. Step 8

    Consume from Kafka and land in Snowflake

    Use Kafka Connect (Snowflake Connector) or Python Kafka consumer to read vbak-source-topic and upsert into Snowflake. Use VBELN (document number) + MANDT as the upsert key.

    Verify
    Consumer runs and rows appear in Snowflake VBAK table in real-time.
  9. Step 9

    Monitor SLT replication lag

    In LTRS, monitor the delta queue. Confirm replication lag is within your SLA (e.g., < 5 minutes).

    Do this in SAP
    Open transaction LTRS (opens in new tab) and navigate to Monitor → Replication Status.
    Verify
    LTRS shows green status, replication lag < SLA.
  10. Step 10

    Handle multi-Z-field scenarios

    At scale, VBAK may have 5+ Z-fields. Confirm SLT includes all in the Kafka payload. If custom extraction is needed, use parallel SLT jobs per VKORG to avoid contention.

    Verify
    VBAK records in Kafka and Snowflake include all Z-fields (verified via SELECT *)

Troubleshooting

SLT replication lag exceeds SLA

SLT processes sequentially. Parallelize by creating separate replication objects per VKORG or company code. Scale Kafka brokers. Monitor network bandwidth.

Snowflake upsert is slow

Kafka Snowflake connector batches rows; adjust batch size and commit interval. Use MERGE statements instead of individual INSERTs.
