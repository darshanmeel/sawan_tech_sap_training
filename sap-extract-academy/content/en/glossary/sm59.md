### What is SM59?

`SM59` is the SAP transaction for **RFC Destination Configuration** — the administrative interface where connection parameters for remote systems are defined and stored. In SAP's RFC model, every external system that wants to call a function module on SAP (or every SAP system that wants to call another SAP system) is represented as a named **destination** in SM59. This destination acts as a logical alias: programs reference it by name, and SM59 resolves that name to the actual network address, port, credentials, and protocol settings of the remote system.

For data extraction, SM59 is the foundational configuration layer for any RFC-based connection. Whether you are using Python with the `pyrfc` library, Azure Data Factory's SAP connector, Informatica, Talend, or a custom ABAP program making remote function calls, the connection parameters ultimately resolve through SM59 or its equivalent. Understanding SM59 — what it contains, how it is structured, and how to test it — is essential for troubleshooting connectivity problems, which are among the most common causes of extraction pipeline failures in the early stages of a project.

### How it works

SM59 destinations are categorised by **connection type**, which determines the protocol used for communication. The most common types for data extraction are:

- **Type 3 (ABAP Connection)**: SAP-to-SAP RFC connections, used when one SAP system calls function modules on another.
- **Type TCP/IP (T)**: Used for connections to external non-SAP programs via the SAP RFC library. This is the type used by Python/pyrfc, SAP JCo (Java Connector), and most third-party extraction tools. The external program registers itself with the SAP gateway using the SAP RFC library, and SM59 defines how to reach it — or, in the extraction direction, defines the SAP system endpoint that external tools connect to.
- **Type H (HTTP)**: HTTP-based connections, relevant when SAP calls external REST services. Distinct from SICF, which handles inbound HTTP to SAP.

Within each destination, you configure the **program ID**, **gateway host**, **gateway service** (typically `sapgw00` for instance 00), **target host**, **system number**, **client**, and **logon credentials**. The credentials embedded in SM59 are often a technical user account with restricted authorizations — sufficient to call specific function modules but not to log into the SAP GUI or access sensitive financial reports. Administrators control what that technical user can do through the SAP authorization system.

### Why it matters for data extraction

When a Python script using `pyrfc` fails to connect to SAP, the root cause is almost always one of three things: the SM59 destination does not exist, the destination exists but the network connection is blocked (firewall, wrong port), or the credentials are incorrect or the user is locked. SM59 is the first place to check in all three scenarios. The transaction provides a **Connection Test** button that executes a live test of the configured connection, reporting success or failure with an error code. This single button eliminates hours of trial-and-error network debugging.

SM59 also matters for understanding the SAP system landscape. Large enterprises run multiple SAP systems — development, quality, and production, sometimes across multiple landscapes (ECC, S/4HANA, BW). Each has its own SM59 configuration, and destinations defined in a development system do not automatically exist in production. When a pipeline that works in development fails in production, missing or misconfigured SM59 destinations are a frequent culprit. Extraction pipeline documentation should always include the SM59 destination name and the minimum set of authorizations required for the extraction user, so that basis administrators can replicate the configuration across environments.

### Common pitfalls

The most frequent SM59-related extraction problem is **credential expiry**. SAP user accounts configured in SM59 have password expiration policies like any other SAP user. When the password expires, all RFC calls from external tools fail with an authorization error that is not always clearly labeled as a password expiry. The fix is simple — reset the password and update SM59 — but the diagnosis requires knowing where to look. Production extractions should use service accounts with `Password change not required` flag enabled, and password changes should be coordinated with the teams managing SM59 destinations.

Another pitfall is the **SAP message server vs. application server confusion**. Some extraction tools support load-balanced connections through the SAP message server (which distributes connections across application server instances), while others require a direct connection to a specific application server. Configuring an SM59 destination with the wrong address type can cause intermittent connectivity — the pipeline works when it hits the right application server and fails when load balancing sends it to a server without the necessary RFC gateway configuration. Always confirm with the basis team whether to use message server or application server addressing.

### In practice

A data engineering team is setting up a Python/pyrfc pipeline to extract `VBAK` and `VBAP` from a production SAP ECC system. They have written and tested the Python code against the development system using destination name `SAP_DEV`. When they try to run the same code against production (`SAP_PROD`), connection fails immediately. They open SM59 on the production system and find that `SAP_PROD` does not exist — the basis team created the destination in development but did not transport it to production (SM59 destinations are not transported via the standard change management system; they must be manually recreated or exported/imported). The basis team creates the destination in production, runs the Connection Test in SM59 (success), and the Python pipeline connects immediately. The data engineering team adds SM59 destination verification to their pre-go-live checklist for all future environment migrations.
